# 13 商用環境構築手順書（本番デプロイ）

クラウド上に本番環境を初回構築する手順。
ローカル開発環境の構築は [12_テスト環境構築手順書.md](12_テスト環境構築手順書.md) を参照すること。
運用・再デプロイ手順は [14_運用マニュアル.md](14_運用マニュアル.md) を参照すること。

---

## 1. 前提条件

### 1.1 必要なアカウント

以下をすべて作成済みであること。

| サービス | URL | 用途 | プラン |
|---|---|---|---|
| GitHub | github.com | ソースコード管理・CI/CD | 無料 |
| Supabase | supabase.com | PostgreSQL + pgvector | Free Tier |
| Render | render.com | FastAPI ホスティング | Free |
| Vercel | vercel.com | Next.js ホスティング | Hobby（無料） |
| xAI | console.x.ai | Grok API | 従量課金 |
| OpenAI | platform.openai.com | Embedding API | 従量課金 |

### 1.2 必要なソース

- `main` ブランチにデプロイ可能なコードがプッシュされていること
- テストが全件 PASS していること（`pytest tests/ -v`）

### 1.3 APIキーの事前取得

デプロイ前に以下のAPIキーを取得しておくこと。

#### xAI（Grok API）

1. [console.x.ai](https://console.x.ai) にサインインする
2. 左メニューの「API Keys」→「Create API Key」をクリック
3. Key name を入力（例: `rag-chatbot-prod`）し「Create」
4. 表示された `xai-xxx...` の文字列をコピーし、安全な場所に保存する

> **このキーは作成直後にしか確認できない。必ずコピーすること。**

#### OpenAI（Embedding API）

1. [platform.openai.com](https://platform.openai.com) にサインインする
2. 右上のアカウントアイコン →「API keys」を選択
3. 「Create new secret key」をクリック
4. Name を入力（例: `rag-chatbot-prod`）し「Create secret key」
5. 表示された `sk-xxx...` の文字列をコピーし、安全な場所に保存する

> **このキーは作成直後にしか確認できない。必ずコピーすること。**

## 2. GitHub リポジトリのセットアップ

### 2.1 リポジトリ作成

1. GitHub で新規リポジトリを作成する（例: `rag-chatbot`）
2. Visibility: **Public**（ポートフォリオとして公開する場合）

### 2.2 初回プッシュ

```bash
git remote add origin https://github.com/<username>/rag-chatbot.git
git push -u origin main
```

## 3. Supabase 本番プロジェクトのセットアップ

### 3.1 プロジェクト作成

1. [supabase.com](https://supabase.com) にサインインし「New Project」を作成する
2. プロジェクト名: `rag-chatbot-prod`
3. **Region**: `Northeast Asia (Tokyo)` または最寄りリージョンを選択
4. Database Password: 強力なパスワードを設定し **必ずメモする**

### 3.2 スキーマの作成

SQL Editor を開き、以下を順番に実行する。

```sql
-- pgvector 拡張の有効化
CREATE EXTENSION IF NOT EXISTS vector;

-- documents テーブル
CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    source_url      TEXT NOT NULL UNIQUE,
    file_type       TEXT NOT NULL CHECK (file_type IN ('markdown', 'pdf')),
    organization    TEXT,
    authority_score NUMERIC(3, 2) NOT NULL DEFAULT 0.50 CHECK (authority_score BETWEEN 0 AND 1),
    content_date    DATE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- document_chunks テーブル
CREATE TABLE document_chunks (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id   UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index   INTEGER NOT NULL,
    content       TEXT NOT NULL,
    embedding     VECTOR(1536),
    token_count   INTEGER,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);

-- chat_history テーブル
CREATE TABLE chat_history (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id  TEXT NOT NULL,
    query       TEXT NOT NULL,
    response    TEXT NOT NULL,
    sources     JSONB NOT NULL DEFAULT '[]',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- updated_at 自動更新トリガー
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_documents_updated_at
BEFORE UPDATE ON documents
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- インデックス
CREATE INDEX idx_documents_organization ON documents (organization);
CREATE INDEX idx_chat_history_session_id ON chat_history (session_id);
CREATE INDEX idx_chat_history_created_at ON chat_history (created_at);
```

> **ベクトル検索インデックスは Ingest バッチ実行後に作成する（§7参照）。**
> データが 0 件の状態でインデックスを作成しても意味がない。

### 3.3 接続情報の取得

Settings → Database → **Connection string (URI)** をコピーし、後続手順で使用する。

```
postgresql://postgres:[password]@[host]:5432/postgres
```

## 4. Render（バックエンド）のデプロイ

### 4.1 Web Service の作成

1. [render.com](https://render.com) にサインインし「New Web Service」を選択
2. GitHub リポジトリを連携し `rag-chatbot` を選択

### 4.2 サービス設定

| 設定項目 | 値 |
|---|---|
| Name | `rag-chatbot-api` |
| Region | `Oregon (US West)` |
| Branch | `main` |
| Root Directory | `backend` |
| Runtime | `Python 3.12` |
| Build Command | `pip install -r requirements.txt` |
| Start Command | `uvicorn app.main:app --host 0.0.0.0 --port $PORT` |
| Instance Type | `Free` |

### 4.3 環境変数の設定

Render ダッシュボードの「Environment」タブで以下を設定する。

| 変数名 | 値 | 備考 |
|---|---|---|
| `DATABASE_URL` | `postgresql://...`（§3.3で取得） | Supabase 本番 URI |
| `GROK_API_KEY` | `xai-xxx` | xAI コンソールで取得 |
| `OPENAI_API_KEY` | `sk-xxx` | OpenAI プラットフォームで取得 |
| `LLM_PROVIDER` | `grok` | |
| `TOP_K_DEFAULT` | `5` | |
| `CORS_ORIGINS` | `http://localhost:3000` | §6 で本番 Vercel URL に更新する |

> `CORS_ORIGINS` はこの時点では仮値を設定しておき、§6（Vercel デプロイ後）に本番 URL へ更新する。

### 4.4 デプロイの確認

「Create Web Service」→ ビルドログを確認する。
デプロイ完了後:

```bash
curl https://<app-name>.onrender.com/health
# 期待: {"status":"ok","version":"0.1.0","services":{"database":"ok","llm":"ok"}}
```

## 5. Vercel（フロントエンド）のデプロイ

### 5.1 プロジェクトの作成

1. [vercel.com](https://vercel.com) にサインインし「Add New Project」を選択
2. GitHub リポジトリを連携し `rag-chatbot` をインポート

### 5.2 ビルド設定

| 設定項目 | 値 |
|---|---|
| Framework Preset | Next.js（自動検出） |
| Root Directory | `frontend` |
| Build Command | `npm run build`（デフォルト） |
| Output Directory | `.next`（デフォルト） |

### 5.3 環境変数の設定

「Environment Variables」セクションで以下を設定する。

| 変数名 | 値 | 備考 |
|---|---|---|
| `NEXT_PUBLIC_API_BASE_URL` | `https://<app-name>.onrender.com` | §4 で確定した Render URL |

### 5.4 デプロイの確認

「Deploy」をクリックし、デプロイが成功したら表示された URL にアクセスしてチャット画面が表示されることを確認する。

---

## 6. CORS の設定を更新

Vercel の URL が確定したら Render の `CORS_ORIGINS` を更新する。

1. Render ダッシュボード → `rag-chatbot-api` → Environment
2. `CORS_ORIGINS` を以下のように更新する

```
https://<project-name>.vercel.app,http://localhost:3000
```

3. 「Save Changes」→ 自動で再デプロイされる

---

## 7. 本番データの投入（Ingest バッチ）

### 7.1 本番用 .env の準備

```bash
cp .env .env.prod
```

`.env.prod` の `DATABASE_URL` を本番 Supabase の接続文字列に変更する。

### 7.2 バッチ実行

```bash
docker compose --env-file .env.prod run --rm ingest
```

### 7.3 ベクトル検索インデックスの作成

全チャンクを insert した後、Supabase SQL Editor で以下を実行する。

```sql
-- lists は sqrt(チャンク総数) 程度が推奨
-- 例: 1000 チャンク → lists=32、1万チャンク → lists=100
CREATE INDEX idx_document_chunks_embedding
ON document_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 10);
```

## 8. 動作確認チェックリスト

| # | 確認項目 | 方法 |
|---|---|---|
| 1 | バックエンド API が起動している | `GET /health` → status: ok |
| 2 | DB 接続が正常 | `/health` の `services.database` が `"ok"` |
| 3 | LLM 接続が正常 | `/health` の `services.llm` が `"ok"` |
| 4 | チャット画面が表示される | Vercel URL にアクセス |
| 5 | チャットが動作する | 質問を入力して回答と出典が表示される |
| 6 | CORS エラーが出ない | ブラウザの DevTools でネットワークエラーがない |
| 7 | データが取り込まれている | チャットで投入データに関する質問に回答できる |

## 9. GitHub Actions による CI/CD（任意）

`main` ブランチへのプッシュ時にテストを自動実行するワークフロー。

```yaml
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - name: Install dependencies
        run: pip install -r backend/requirements.txt
      - name: Run tests
        run: |
          cd backend
          pytest tests/ -v --tb=short
```

GitHub Actions によるテスト通過を確認してから Render・Vercel の自動デプロイが走る。

## 10. Render Free Tier の注意事項

| 制限 | 内容 | 対策 |
|---|---|---|
| スリープ | 15 分無アクセスでインスタンスが停止 | 初回リクエスト時に 30〜50 秒の起動時間が発生。フロントエンドに「起動中...」の表示を追加する |
| CPU | 0.1 vCPU / 512 MB RAM | 同時リクエストが多い場合は有料プランを検討 |
| 稼働時間 | 750 時間/月 | 1 インスタンスなら常時稼働可能 |

---

## 11. Supabase Free Tier の注意事項

| 制限 | 内容 | 対策 |
|---|---|---|
| DB サイズ | 500 MB | チャンク数を増やしすぎないよう注意 |
| 非アクティブ停止 | 1 週間アクセスがないとプロジェクトが一時停止 | Supabase ダッシュボードから「Resume project」で再起動 |
| 帯域幅 | 2 GB/月 | ポートフォリオ用途では十分 |
