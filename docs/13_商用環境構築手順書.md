# 13 商用環境構築手順書（GCP本番デプロイ - Terraform版）

Terraform (Infrastructure as Code) を使用して、Google Cloud 上に検証環境（Staging）および本番環境（Production）を構築する手順です。
ローカル開発環境は [12_テスト環境構築手順書.md](12_テスト環境構築手順書.md)、構築後の運用は [15_運用マニュアル.md](15_運用マニュアル.md) を参照してください。

> [!IMPORTANT]
> プロジェクト内のファイルには環境固有の絶対パスを使用しません。`terraform/`、`backend/`、`frontend/`、`db/` があるディレクトリをプロジェクトルートとします。外部ツールのインストール先には `C:\apps\` を例示するため、異なる場所へインストールした場合は実際のパスへ読み替えてください。

---

## 1. 構築する環境

セキュリティ、課金集計、誤操作防止のため、検証環境と本番環境は別の GCP プロジェクトに分離します。

| 環境 | 用途 | プロジェクトIDの例 | Terraform workspace |
|---|---|---|---|
| Staging | デプロイ確認、受入テスト | `your-rag-chatbot-stg` | `stg` |
| Production | 一般公開、本番運用 | `your-rag-chatbot-prod` | `prod` |

プロジェクトIDは全世界で一意であり、作成後は変更できません。表の値は例なので、実際には利用可能なIDを決めてください。

> [!CAUTION]
> 現在のTerraformはポートフォリオ・検証用途を前提としており、Cloud SQLの削除保護をStaging・Productionの両方で無効にしています。本番公開前に§14の対策を実施してください。

構築される主なリソースは以下です。

- Artifact Registry（Dockerイメージ）
- Cloud SQL for PostgreSQL 15（pgvector）
- Cloud Run（FastAPIバックエンド、Next.jsフロントエンド）
- アプリケーション用サービスアカウント
- Vertex AI / Cloud SQL接続用IAMロール

## 2. 前提条件

### 2.1 必要なアカウント

- Googleアカウント
- 有効な Google Cloud 請求先アカウント
- 対象プロジェクトを作成できる権限
- 請求先アカウントをプロジェクトへ関連付けられる権限

個人のポートフォリオ環境では、プロジェクト作成者がOwnerであれば後続作業を実行できます。組織管理下では、§4.4のIAM権限を管理者へ依頼してください。

### 2.2 必要なローカルツール

| ツール | 確認コマンド | 用途 |
|---|---|---|
| Google Cloud CLI | `gcloud version` | 認証、プロジェクト操作、Cloud Build |
| Terraform 1.0以上 | `terraform version` | GCPリソース構築 |
| Docker Desktop | `docker info` | ローカル確認、DB初期化、Ingest |
| Cloud SQL Auth Proxy v2（任意） | `& "C:\apps\cloud-sql-proxy\cloud-sql-proxy.exe" --version` | ローカルPC上のpsqlからCloud SQLへ接続する場合のみ使用 |
| Git | `git --version` | ソースコード管理 |

インストール先は以下です。

- [Google Cloud CLIのインストール](https://cloud.google.com/sdk/docs/install)
- [Terraformのインストール](https://developer.hashicorp.com/terraform/install)
- [Cloud SQL Auth Proxyのインストール](https://cloud.google.com/sql/docs/postgres/connect-auth-proxy#install-proxy)

GCP Ingestでは§8のDocker版Proxyを使用するため、Windows版Proxyのインストールは不要です。ローカルPC上のpsqlから接続する場合は、Windows 64-bit版を次の例でインストールできます。

```powershell
$CLOUD_SQL_PROXY_DIR = "C:\apps\cloud-sql-proxy"
$CLOUD_SQL_PROXY_EXE = Join-Path $CLOUD_SQL_PROXY_DIR "cloud-sql-proxy.exe"

New-Item -ItemType Directory -Force $CLOUD_SQL_PROXY_DIR
curl.exe -L "https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.22.1/cloud-sql-proxy.x64.exe" -o $CLOUD_SQL_PROXY_EXE
& $CLOUD_SQL_PROXY_EXE --version
```

既にGoogle Cloud CLIを導入済みの場合は、古いバージョンのまま使わないよう更新します。

> [!IMPORTANT]
> Windows PowerShell 5.1では、`-var-file=...`、`-target=...`、`-out=...` のようなTerraform引数を、引数全体ごと引用符で囲む必要があります。本書のPowerShell例はこの形式で記載します。

```powershell
gcloud components update
gcloud version
```

## 3. Google Cloudのアカウント・課金準備

### 3.1 Google Cloudコンソールへログイン

1. [Google Cloudコンソール](https://console.cloud.google.com/) を開く。
2. デプロイに使用するGoogleアカウントでログインする。
3. 初回利用時は利用規約への同意と支払い情報の登録を完了する。

### 3.2 請求先アカウントの作成・確認

1. Google Cloudコンソール左上のナビゲーションメニューを開く。
2. **お支払い（Billing）**を開く。
3. 請求先アカウントがない場合は、画面の案内に従って作成する。
4. 請求先アカウントが有効であることを確認する。

CLIで利用可能な請求先アカウントを確認する場合は次を実行します。

```powershell
gcloud auth login
gcloud billing accounts list --filter="open=true"
```

表示された `ACCOUNT_ID` を後続手順で使用します。請求先アカウントIDや支払い情報はリポジトリへ記録しないでください。

### 3.3 予算アラートの作成（推奨）

想定外の利用を早期に検知するため、プロジェクト作成後に環境ごとの予算アラートを設定します。

1. Google Cloudコンソールで **お支払い** → **予算とアラート**を開く。
2. **予算を作成**を選択する。
3. 対象プロジェクトを指定する。
4. 月額予算と通知しきい値（例: 50%、90%、100%）を設定する。
5. 通知先メールを確認して保存する。

> [!NOTE]
> 予算アラートは通知機能であり、利用料金を自動停止する機能ではありません。

## 4. GCPプロジェクトの作成

### 4.1 プロジェクトIDを決める

次の条件を満たすIDを、StagingとProductionで1つずつ決めます。

- 6～30文字
- 小文字、数字、ハイフンのみ
- 先頭は英字
- 全世界で一意
- 個人名、メールアドレス、秘密情報を含めない

以降は次の変数名で表します。

| 変数 | 内容 |
|---|---|
| `STG_PROJECT_ID` | Stagingの実際のプロジェクトID |
| `PROD_PROJECT_ID` | Productionの実際のプロジェクトID |
| `BILLING_ACCOUNT_ID` | 有効な請求先アカウントID |

### 4.2 Google Cloudコンソールで作成する場合

StagingとProductionについて、それぞれ次を実施します。

1. Google Cloudコンソール上部のプロジェクト選択欄を開く。
2. **新しいプロジェクト**を選択する。
3. プロジェクト名を入力する（例: `RAG Chatbot Staging`）。
4. 一意なプロジェクトIDを入力する。
5. 組織を利用している場合は、所属先の組織またはフォルダを選択する。
6. **作成**を選択する。
7. **お支払い** → **アカウント管理**から、有効な請求先アカウントがリンクされていることを確認する。

### 4.3 gcloudで作成する場合

以下はPowerShellの例です。`Read-Host`の実行後に入力欄が表示されたら、§4.1で決めた実際の値を入力してEnterキーを押します。引用符内の文言は入力を促すメッセージであり、設定値ではありません。

```powershell
$STG_PROJECT_ID = Read-Host "Stagingの実プロジェクトID"
$PROD_PROJECT_ID = Read-Host "Productionの実プロジェクトID"
$BILLING_ACCOUNT_ID = Read-Host "請求先アカウントID"

$PROJECT_ID_PATTERN = "^[a-z][a-z0-9-]{4,28}[a-z0-9]$"
if ($STG_PROJECT_ID -notmatch $PROJECT_ID_PATTERN) {
  throw "Stagingに、形式を満たす実際のプロジェクトIDを指定してください。"
}
if ($PROD_PROJECT_ID -notmatch $PROJECT_ID_PATTERN) {
  throw "Productionに、形式を満たす実際のプロジェクトIDを指定してください。"
}

gcloud projects create $STG_PROJECT_ID --name="RAG Chatbot Staging"
gcloud projects create $PROD_PROJECT_ID --name="RAG Chatbot Production"

gcloud billing projects link $STG_PROJECT_ID --billing-account=$BILLING_ACCOUNT_ID
gcloud billing projects link $PROD_PROJECT_ID --billing-account=$BILLING_ACCOUNT_ID
```

組織またはフォルダ配下に作成する場合は、`gcloud projects create`へ `--organization=[ORGANIZATION_ID]` または `--folder=[FOLDER_ID]` を追加します。

作成・課金状態を確認します。

```powershell
gcloud projects describe $STG_PROJECT_ID
gcloud billing projects describe $STG_PROJECT_ID

gcloud projects describe $PROD_PROJECT_ID
gcloud billing projects describe $PROD_PROJECT_ID
```

### 4.4 構築担当者に必要なIAM権限

個人プロジェクトでプロジェクト作成者がOwnerの場合、通常は追加設定不要です。組織管理下で最小権限を付与する場合は、管理者から構築担当者へ次のロールが必要です。

| ロール | ロールID | 用途 |
|---|---|---|
| Service Usage管理者 | `roles/serviceusage.serviceUsageAdmin` | API有効化 |
| Service Usage Consumer | `roles/serviceusage.serviceUsageConsumer` | ADCのクォータプロジェクト利用 |
| Artifact Registry管理者 | `roles/artifactregistry.admin` | Dockerリポジトリ作成 |
| Cloud SQL管理者 | `roles/cloudsql.admin` | DBインスタンス、DB、ユーザー作成 |
| Cloud Run管理者 | `roles/run.admin` | Cloud Runサービス作成 |
| サービスアカウント管理者 | `roles/iam.serviceAccountAdmin` | アプリ用サービスアカウント作成 |
| サービスアカウントユーザー | `roles/iam.serviceAccountUser` | Cloud Runへのサービスアカウント設定 |
| Project IAM管理者 | `roles/resourcemanager.projectIamAdmin` | IAMロール割当 |
| Cloud Build編集者 | `roles/cloudbuild.builds.editor` | ビルド実行 |

権限はStagingとProductionの両プロジェクトへ個別に付与します。サービスアカウントのJSONキーは作成せず、ローカルはADC、Cloud Runはアタッチしたサービスアカウントを使用します。

## 5. gcloudの環境別設定と認証

### 5.1 gcloud configurationを分離する

誤ったプロジェクトへデプロイしないよう、環境ごとにconfigurationを作成します。

> [!IMPORTANT]
> configuration名（例: `rag-chatbot-stg`）はローカルPC内の設定名であり、GCPプロジェクトIDではありません。ADC、Terraform、課金設定には、`gcloud config get-value project`で表示される実プロジェクトIDを使用します。

PowerShellを開き直した場合は、実際のプロジェクトIDを変数へ設定し直します。

```powershell
$STG_PROJECT_ID = Read-Host "Stagingの実プロジェクトID"
$PROD_PROJECT_ID = Read-Host "Productionの実プロジェクトID"

$PROJECT_ID_PATTERN = "^[a-z][a-z0-9-]{4,28}[a-z0-9]$"
if ($STG_PROJECT_ID -notmatch $PROJECT_ID_PATTERN) {
  throw "Stagingに、形式を満たす実際のプロジェクトIDを指定してください。"
}
if ($PROD_PROJECT_ID -notmatch $PROJECT_ID_PATTERN) {
  throw "Productionに、形式を満たす実際のプロジェクトIDを指定してください。"
}

# configurationの作成は初回のみ
gcloud config configurations create rag-chatbot-stg
gcloud config set project $STG_PROJECT_ID
gcloud config set run/region asia-northeast1

gcloud config configurations create rag-chatbot-prod
gcloud config set project $PROD_PROJECT_ID
gcloud config set run/region asia-northeast1
```

configurationを作成済みの場合は、`create`を再実行せず、次の手順で対象configurationを有効化します。

作業前に対象環境を選択し、必ず確認します。

```powershell
# Stagingを操作する場合
gcloud config configurations activate rag-chatbot-stg

# Productionを操作する場合は prod を指定
# gcloud config configurations activate rag-chatbot-prod

gcloud config get-value project
gcloud auth list
```

### 5.2 Terraform用ADCを作成する

`gcloud auth login`の認証情報とTerraformが使用するApplication Default Credentials（ADC）は別です。
対象configurationを有効化した状態で、両方に同じGoogleアカウントを使用します。

```powershell
gcloud auth login
gcloud auth application-default login

$PROJECT_ID = gcloud config get-value project
if ($PROJECT_ID -notmatch "^[a-z][a-z0-9-]{4,28}[a-z0-9]$") {
  throw "gcloudに実際のプロジェクトIDが設定されていません。§5.1を確認してください。"
}

gcloud projects describe $PROJECT_ID --format="value(projectId)" | Out-Null
if ($LASTEXITCODE -ne 0) {
  throw "対象プロジェクトが存在しないか、アクセス権がありません。"
}

gcloud auth application-default set-quota-project $PROJECT_ID
if ($LASTEXITCODE -ne 0) {
  throw "ADCのクォータプロジェクトを設定できませんでした。プロジェクトIDとIAM権限を確認してください。"
}

# WindowsでADCファイルが作成されたことを確認
Test-Path "$env:APPDATA\gcloud\application_default_credentials.json"

# ADCでアクセストークンを取得できることを確認
gcloud auth application-default print-access-token | Out-Null
if ($LASTEXITCODE -ne 0) {
  throw "ADCでアクセストークンを取得できませんでした。"
}
```

`Test-Path`が `True` となり、最後までエラーなく完了することを確認します。失敗した場合はTerraformを実行しません。

`serviceusage.services.use` 権限のエラーが表示された場合は、最初に `$PROJECT_ID` が実際の対象プロジェクトIDであることを確認します。正しい場合は、ADCで使用したアカウントに対象プロジェクトの `roles/serviceusage.serviceUsageConsumer` が必要です。組織管理下では、§4.4の権限を管理者へ依頼してください。

自動で開いたブラウザで `cloud-platform scope is required but not consented` が表示された場合は、localhostへの自動リダイレクトを使わない次の方法でやり直します。

```powershell
gcloud auth application-default login --no-launch-browser
```

1. ターミナルに表示されたURLをブラウザで開く。
2. `gcloud`で使用しているものと同じGoogleアカウントを選択する。
3. Google Cloudのデータを操作する権限を含む、表示されたすべての要求権限を確認して許可する。
4. 表示された認証コードをターミナルへ貼り付ける。
5. ADCファイルが作成されたことを再確認する。

```powershell
Test-Path "$env:APPDATA\gcloud\application_default_credentials.json"
gcloud auth application-default set-quota-project $PROJECT_ID
gcloud auth application-default print-access-token | Out-Null
```

Productionへ切り替えた場合も、対象configurationを有効化した後に `$PROJECT_ID = gcloud config get-value project` を再実行し、`set-quota-project`で現在の対象プロジェクトを設定します。

### 5.3 必要API

Terraformは次のAPIを自動的に有効化します。

- Cloud Run Admin API: `run.googleapis.com`
- Cloud SQL Admin API: `sqladmin.googleapis.com`
- Artifact Registry API: `artifactregistry.googleapis.com`
- Cloud Build API: `cloudbuild.googleapis.com`
- Vertex AI API: `aiplatform.googleapis.com`

手動で先に有効化する場合は、対象configurationを確認してから実行します。

```powershell
gcloud services enable run.googleapis.com sqladmin.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com aiplatform.googleapis.com

gcloud services list --enabled --filter="name:(run.googleapis.com OR sqladmin.googleapis.com OR artifactregistry.googleapis.com OR cloudbuild.googleapis.com OR aiplatform.googleapis.com)"
```

## 6. Terraform環境設定

### 6.1 実行場所

プロジェクトルートから `terraform/` へ移動します。

```powershell
cd terraform
```

以降のTerraformコマンドは `terraform/` で実行します。

### 6.2 環境別workspaceを作成する

StagingとProductionで同じstateを共有すると、別環境のリソースを更新・削除する危険があります。必ずworkspaceを分けます。

```powershell
terraform init

# 初回のみ
terraform workspace new stg
terraform workspace new prod

# 今回構築する環境を選択
# Stagingの場合
terraform workspace select stg

# Productionの場合
# terraform workspace select prod

terraform workspace show
```

Productionを構築する場合は `terraform workspace select prod` を使用します。`default` workspaceではデプロイしないでください。

### 6.3 tfvarsを作成する

`terraform/stg.tfvars` と `terraform/prod.tfvars` を作成します。これらは `.gitignore` の `*.tfvars` で除外されます。

`stg.tfvars` の例:

```hcl
project_id   = "your-unique-rag-stg"
region       = "asia-northeast1"
environment  = "stg"
db_password  = "REPLACE_WITH_32_CHAR_URI_SAFE_PASSWORD"
cors_origins = "*"
db_tier      = "db-f1-micro"
```

`prod.tfvars` の例:

```hcl
project_id   = "your-unique-rag-prod"
region       = "asia-northeast1"
environment  = "prod"
db_password  = "REPLACE_WITH_32_CHAR_URI_SAFE_PASSWORD"
cors_origins = "*"
db_tier      = "db-custom-1-3840"
```

初回はフロントエンドURLが未確定なので `cors_origins = "*"` とし、デプロイ後に§9.1で対象環境のフロントエンドURLへ制限します。

`db-f1-micro`は共有コアの低コスト構成であり、可用性SLAの対象外です。Productionでは上記の専用コア構成を出発点とし、負荷試験と可用性要件に基づいてtierを決定してください。ただし、専用コアへ変更するだけではSLA対象にならず、高可用性（HA）構成も必要です。

> [!IMPORTANT]
> 現在のTerraformはDBパスワードを接続URLへ直接埋め込みます。URL予約文字による接続エラーを避けるため、十分に長い英大文字・英小文字・数字のみのランダム値を使用してください。tfvars、Terraform state、planファイルをGitへコミットしないでください。

### 6.4 設定を検証する

```powershell
$TFVARS = "$(terraform workspace show).tfvars"
if (-not (Test-Path -LiteralPath $TFVARS)) {
  throw "$TFVARS が見つかりません。§6.3を確認してください。"
}

terraform fmt -check
if ($LASTEXITCODE -ne 0) {
  throw "Terraformファイルの書式に差分があります。terraform fmtを実行してください。"
}

terraform validate
if ($LASTEXITCODE -ne 0) {
  throw "Terraform設定の検証に失敗しました。"
}

terraform workspace show
gcloud config get-value project

# サンプル値が残っていないことを確認（何も表示されないこと）
Select-String -Path $TFVARS -Pattern 'your-unique-rag-(stg|prod)|REPLACE_WITH'
```

`terraform workspace show`とtfvarsの `environment`、`gcloud config get-value project`とtfvarsの `project_id` が一致していることを目視確認します。`Select-String`で行が表示された場合は、サンプル値が残っているため `terraform apply` を実行しないでください。

## 7. インフラ構築と初回デプロイ

Cloud Runは参照するコンテナイメージがArtifact Registryに存在しないと作成できないため、初回は段階的に適用します。

以下はStagingの例です。Productionを操作する場合は、Stagingの3行をコメントアウトし、Productionの3行を有効にします。

```powershell
# Stagingの場合
gcloud config configurations activate rag-chatbot-stg
terraform workspace select stg
$TFVARS = "stg.tfvars"

# Productionの場合
# gcloud config configurations activate rag-chatbot-prod
# terraform workspace select prod
# $TFVARS = "prod.tfvars"

$PROJECT_ID = gcloud config get-value project
$REGION = "asia-northeast1"
$IMAGE_BASE = "${REGION}-docker.pkg.dev/${PROJECT_ID}/rag-chatbot-images"
```

### 7.1 API・Artifact Registry・Cloud SQL・IAMを作成する

適用前にADCを確認します。`False`が表示された場合やアクセストークンを取得できない場合は、§5.2の認証を完了してから進みます。

```powershell
if (-not (Test-Path "$env:APPDATA\gcloud\application_default_credentials.json")) {
  throw "ADCファイルが見つかりません。§5.2を確認してください。"
}

gcloud auth application-default print-access-token | Out-Null
if ($LASTEXITCODE -ne 0) {
  throw "ADCでアクセストークンを取得できませんでした。§5.2を確認してください。"
}

terraform apply "-var-file=$TFVARS" "-target=google_project_service.enabled_apis" "-target=google_artifact_registry_repository.app_repo" "-target=google_sql_database_instance.db_instance" "-target=google_sql_database.database" "-target=google_sql_user.db_user" "-target=google_service_account.app_sa" "-target=google_project_iam_member.vertex_ai_user" "-target=google_project_iam_member.cloud_sql_client"
```

確認画面で対象プロジェクトと作成リソースを確認し、問題がなければ `yes` を入力します。

> [!NOTE]
> Terraformの `-target` は初回の循環依存を解消するためだけに使用します。通常更新では§9.2の全体適用を使用してください。

### 7.2 Artifact RegistryへのDocker認証を設定する

```powershell
gcloud auth configure-docker "${REGION}-docker.pkg.dev"
gcloud artifacts repositories describe rag-chatbot-images --location=$REGION
```

### 7.3 バックエンドイメージをビルド・プッシュする

`terraform/`で実行します。

```powershell
gcloud builds submit --tag "${IMAGE_BASE}/backend:latest" ../backend
```

`backend/.gcloudignore` と `backend/.dockerignore` により、仮想環境、テストキャッシュ、環境変数ファイルなどを送信対象から除外します。これらは依存パッケージや実行環境のバージョンを変更するファイルではありません。

ビルドが `SUCCESS` になったことを確認し、Artifact Registryに `backend:latest` が存在することを確認します。

```powershell
gcloud artifacts docker images list `
  "${REGION}-docker.pkg.dev/${PROJECT_ID}/rag-chatbot-images" `
  --include-tags `
  --filter="package~'/backend$'"
```

何も表示されない場合はCloud Runをapplyせず、Cloud Buildの失敗原因を解消してから再ビルドしてください。

Artifact RegistryのリポジトリIDはTerraform実装に合わせて `rag-chatbot-images` です。

### 7.4 バックエンドを先行デプロイする

```powershell
terraform apply "-var-file=$TFVARS" "-target=google_cloud_run_v2_service.backend" "-target=google_cloud_run_v2_service_iam_member.backend_public"

$BACKEND_URL = terraform output -raw backend_url
$BACKEND_URL
```

URLが表示されたら、Cloud Runコンソールでもバックエンドサービスが作成されていることを確認します。`/health` のDB判定が正常でも、必須テーブルの存在は保証されないため、§8のDB初期化を必ず実行します。

### 7.5 フロントエンドイメージをビルド・プッシュする

Next.jsへバックエンドURLをビルド時に埋め込みます。

```powershell
gcloud builds submit --config=../frontend/cloudbuild.yaml --substitutions="_BACKEND_URL=$BACKEND_URL,_PROJECT_ID=$PROJECT_ID" ../frontend
```

`frontend/.gcloudignore` は `node_modules`、`.next`、ローカルの環境変数ファイルなどをCloud Buildのアップロード対象から除外します。この対応でNode.jsのバージョンは変更していません。使用バージョンは従来どおり `frontend/Dockerfile` の `node:18-alpine` です。

### 7.6 全リソースを適用する

```powershell
$PLAN_FILE = "${env:COMPUTERNAME}-${TFVARS}.tfplan"

  terraform plan "-var-file=$TFVARS" "-out=$PLAN_FILE"
```

```powershell

$BACKEND_URL = terraform output -raw backend_url
```

Terraformのoutputが見つからない場合は、Cloud Runがまだ作成されていないか、対象workspaceのstateへ反映されていません。対象プロジェクト上のCloud Runサービスを直接確認する場合は、次を実行します。

```powershell
gcloud run services list `
  --project=$PROJECT_ID `
  --region=$REGION `
  --format="table(metadata.name,status.url)"
```

## 8. Cloud SQLの初期化とRAGデータ投入

### 8.1 Docker内でCloud SQL Auth Proxyを起動する

プロジェクトルートへ戻り、対象環境とCloud SQL接続名を設定します。IngestコンテナからWindows上の `host.docker.internal` へ接続する方式は、Docker Desktopのネットワーク設定によって名前解決できない場合があるため使用しません。`docker-compose.gcp.yml` のProxyコンテナをIngestと同じDockerネットワークで起動します。

```powershell
cd ..\

# Stagingの場合
gcloud config configurations activate rag-chatbot-stg
$env:APP_ENV_FILE = ".env.stg"

# Productionの場合
# gcloud config configurations activate rag-chatbot-prod
# $env:APP_ENV_FILE = ".env.prod"

$TERRAFORM_EXE = "C:\apps\terraform_1.15.7\terraform.exe"
if (-not (Test-Path -LiteralPath $TERRAFORM_EXE)) {
  throw "Terraformが見つかりません: $TERRAFORM_EXE"
}

$CONNECTION_NAME_OUTPUT = & $TERRAFORM_EXE "-chdir=terraform" output -raw db_connection_name
if ($LASTEXITCODE -ne 0) {
  throw "TerraformからCloud SQL接続名を取得できません。"
}
$env:CLOUD_SQL_CONNECTION_NAME = $CONNECTION_NAME_OUTPUT.Trim()
$PROJECT_ID = (gcloud config get-value project).Trim()
$REGION = "asia-northeast1"

if ([string]::IsNullOrWhiteSpace($env:CLOUD_SQL_CONNECTION_NAME)) {
  throw "Cloud SQL接続名を取得できません。Terraform workspaceとoutputを確認してください。"
}
if (-not (Test-Path -LiteralPath $env:APP_ENV_FILE)) {
  throw "$env:APP_ENV_FILE が見つかりません。"
}
if (-not (Test-Path -LiteralPath "$env:APPDATA\gcloud\application_default_credentials.json")) {
  throw "ADCが見つかりません。gcloud auth application-default loginを実行してください。"
}

docker compose -f docker-compose.yml -f docker-compose.gcp.yml up -d --force-recreate cloud-sql-proxy
docker compose -f docker-compose.yml -f docker-compose.gcp.yml ps cloud-sql-proxy
docker compose -f docker-compose.yml -f docker-compose.gcp.yml logs --tail 20 cloud-sql-proxy
```

ログに対象インスタンスへの待受開始が表示され、コンテナが起動中であることを確認します。Proxyの5432番ポートはホストへ公開せず、同一Composeネットワーク内だけで使用します。

### 8.2 Dockerのpsqlでスキーマを作成する

Docker Desktopを起動し、`docker info` がエラーなく完了することを確認してください。`DB_PASSWORD`にはtfvarsと同じ値を設定します。

```powershell
$DB_PASSWORD = "YOUR_DATABASE_PASSWORD"

docker compose -f docker-compose.yml -f docker-compose.gcp.yml `
  --profile gcp-tools run --rm --no-deps `
  -e PGPASSWORD=$DB_PASSWORD `
  -v "${PWD}/db:/sql:ro" `
  cloud-sql-psql `
  psql -v ON_ERROR_STOP=1 -h cloud-sql-proxy -p 5432 -U app_user -d rag_chatbot -f /sql/01_schema.sql
```

テーブルを確認します。

```powershell
docker compose -f docker-compose.yml -f docker-compose.gcp.yml `
  --profile gcp-tools run --rm --no-deps `
  -e PGPASSWORD=$DB_PASSWORD `
  cloud-sql-psql `
  psql -h cloud-sql-proxy -p 5432 -U app_user -d rag_chatbot -c "\dt"
```

出力に `documents`、`document_chunks`、`chat_history` の3テーブルがすべて表示されることを確認します。1つでも存在しない場合は、RAGデータ投入やブラウザ確認へ進まず、スキーマSQLのエラーを解消して再実行してください。

Cloud SQL Studioまたはローカルのpsqlを使う方法は、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) も参照してください。

### 8.3 RAGデータを投入する

投入対象を `data/`、メタ情報を `batch/sources.yaml` に用意し、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) に従って実行します。

対象環境の `.env.stg` または `.env.prod` では、DB接続先をComposeサービス名の `cloud-sql-proxy:5432` にします。

```dotenv
DATABASE_URL=postgresql://app_user:YOUR_DB_PASSWORD@cloud-sql-proxy:5432/rag_chatbot
```

Vertex AIを使用してIngestする場合は、ADCファイルをコンテナへ読み取り専用でマウントします。

```powershell
docker compose -f docker-compose.yml -f docker-compose.gcp.yml build ingest

docker compose -f docker-compose.yml -f docker-compose.gcp.yml run --rm --no-deps `
  -v "${env:APPDATA}/gcloud/application_default_credentials.json:/tmp/gcp-adc.json:ro" `
  -e GOOGLE_APPLICATION_CREDENTIALS=/tmp/gcp-adc.json `
  -e USE_VERTEX_AI=true `
  -e GCP_PROJECT_ID=$PROJECT_ID `
  -e GCP_LOCATION=$REGION `
  ingest
```

`batch/ingest.py`、`batch/sources.yaml`、依存パッケージを変更した場合は、必ず上記の `build ingest` を先に実行します。

投入後、必要に応じてRAGデータ投入マニュアルのベクトル検索インデックス再構築・確認手順を実行します。

次のコマンドで登録件数を確認します。

```powershell
docker compose -f docker-compose.yml -f docker-compose.gcp.yml `
  --profile gcp-tools run --rm --no-deps `
  -e PGPASSWORD=$DB_PASSWORD `
  cloud-sql-psql `
  psql -h cloud-sql-proxy -p 5432 -U app_user -d rag_chatbot `
  -c "SELECT (SELECT COUNT(*) FROM documents) AS documents, (SELECT COUNT(*) FROM document_chunks) AS chunks;"
```

`documents` と `chunks` がともに1件以上であることが合格条件です。0件の場合はブラウザ確認へ進まず、`data/`、`batch/sources.yaml`、Ingestログ、ADCを確認して再投入します。

### 8.4 Proxyを終了する

データベース作業が完了したら、Proxyを停止し、セッション用環境変数を削除します。

```powershell
docker compose -f docker-compose.yml -f docker-compose.gcp.yml stop cloud-sql-proxy
Remove-Item Env:APP_ENV_FILE -ErrorAction SilentlyContinue
Remove-Item Env:CLOUD_SQL_CONNECTION_NAME -ErrorAction SilentlyContinue
```

### 9.2 全体を再適用する

```powershell
$WORKSPACE = (terraform workspace show).Trim()
$TFVARS = "$WORKSPACE.tfvars"
$GCLOUD_PROJECT_ID = (gcloud config get-value project).Trim()

if ($WORKSPACE -notin @("stg", "prod")) {
  throw "Terraform workspaceがstgまたはprodではありません。"
}

  throw "$TFVARS が見つかりません。§6.3を確認してください。"
}

$TFVARS_CONTENT = Get-Content -LiteralPath $TFVARS -Raw -Encoding UTF8
$PROJECT_ID_MATCH = [regex]::Match($TFVARS_CONTENT, '(?m)^\s*project_id\s*=\s*"([^"]+)"')
if (-not $PROJECT_ID_MATCH.Success) {
  throw "$TFVARS のproject_idを読み取れません。"
}
$TFVARS_PROJECT_ID = $PROJECT_ID_MATCH.Groups[1].Value

if ($GCLOUD_PROJECT_ID -ne $TFVARS_PROJECT_ID) {
  throw "gcloudのプロジェクト（$GCLOUD_PROJECT_ID）とtfvarsのproject_id（$TFVARS_PROJECT_ID）が一致しません。"
}

$DB_CONNECTION_NAME = terraform output -raw db_connection_name
$STATE_PROJECT_ID = ($DB_CONNECTION_NAME -split ":")[0]
if ($STATE_PROJECT_ID -ne $TFVARS_PROJECT_ID) {
  throw "Terraform stateのプロジェクト（$STATE_PROJECT_ID）とtfvarsのproject_id（$TFVARS_PROJECT_ID）が一致しません。"
}

Write-Host "workspace=$WORKSPACE"
Write-Host "tfvars=$TFVARS"
Write-Host "project_id=$TFVARS_PROJECT_ID"

terraform plan "-var-file=$TFVARS"
terraform apply "-var-file=$TFVARS"
```

## 13. トラブルシューティング

| 症状 | 主な原因 | 対処 |
|---|---|---|
| `terraform` が見つからない | Terraform未インストール、PATH未設定 | §2.2の公式手順でインストール後、ターミナルを開き直す |
| Cloud Billingのクォータ超過でプロジェクトを関連付けられない | 請求先アカウントのプロジェクト関連付け上限 | 不要な関連付けを確認するか、Cloud Billingクォータ増加を申請する |
| ADCへクォータプロジェクトを設定できない | サンプルのプロジェクトID、ADCアカウントの不一致、`serviceusage.services.use` 権限不足 | §5.1で実プロジェクトIDを設定し、§5.2の認証と `roles/serviceusage.serviceUsageConsumer` を確認する |
| ADCで `cloud-platform scope is required but not consented` が表示される | 認証時に必要なスコープが許可されていない | §5.2の `--no-launch-browser` を使用して再認証する |
| API有効化が403 | Service Usage管理者権限不足、課金未設定 | IAMとBillingリンクを確認する |
| Cloud BuildがArtifact Registryへpushできない | Cloud Build実行サービスアカウントの権限不足 | 同一プロジェクトか確認し、必要に応じてArtifact Registry Writerを付与する |
| Cloud Run作成時に `backend:latest not found` が表示される | Artifact Registryが空、バックエンドのCloud Build未実行または失敗 | §7.3でバックエンドをビルド・pushし、イメージの存在確認後に§7.4を再実行する |
| フロントエンドビルドが `COPY ... /app/public` で失敗 | `frontend/public/` が存在しない | リポジトリの `frontend/public/.gitkeep` が存在することを確認する |
| Cloud Buildのアップロードが極端に大きい | `node_modules` や `.next` が送信対象 | `frontend/.gcloudignore` が存在し、ローカル生成物を除外していることを確認する |
| Terraformで `frontend_url` が見つからない | フロントエンドイメージ作成後の全体applyが未実行 | §7.6を実行してCloud Runフロントエンドとoutputをstateへ反映する |
| Cloud Runで `allUsers` を設定できない | 組織ポリシーで公開アクセス禁止 | 組織管理者へ確認し、認証付き公開へ変更する |
| `/health` が `database: error` | DB接続情報不一致、Cloud SQL停止、Cloud SQL権限不足 | Cloud SQL Clientロール、Cloud SQL稼働状態、Cloud Runの`DATABASE_URL`とログを確認する |
| `/health` は正常だがRAG検索に失敗する | `/health` は必須テーブルとデータ件数を検査しない | §8.2で3テーブル、§8.3で文書・チャンク件数を確認する |
| `/health` が `llm: error` | Vertex AI API無効、リージョン・IAM不一致 | API、`GCP_PROJECT_ID`、`GCP_LOCATION`、Vertex AI Userロールを確認する |
| `cloud-sql-proxy` が見つからない | 未インストール、PATH未設定 | §2.2でインストールし、§8.1の `$CLOUD_SQL_PROXY_EXE` を使って起動する |
| PowerShellで `sed` が見つからない | Linux用コマンドをPowerShellで実行 | 文字列加工をせず、`terraform output -raw db_connection_name` から接続名を取得する |
| Docker APIまたは `dockerDesktopLinuxEngine` へ接続できない | Docker Desktopが停止中 | Docker Desktopを起動し、`docker info` の成功後に再実行する |
| `CREATE EXTENSION vector` または `CREATE EXTENSION pgcrypto` が権限エラーになる | 接続ユーザーまたはCloud SQL側の許可拡張・権限が不足 | `app_user`で対象DBへ接続していることを確認し、解消しない場合はCloud SQL管理権限を持つユーザーで拡張を作成してから§8.2を再実行する |
| `relation "document_chunks" does not exist` | `rag_chatbot` DBのスキーマ未作成、または接続先DBの誤り | §8.2で `db/01_schema.sql` を実行し、3テーブルの存在を確認する |
| 画面に「ベクトル検索に失敗しました」と表示される | スキーマ未作成、データ未投入、Embedding・DB検索エラー | Cloud Runバックエンドログを確認する。`UndefinedTableError`なら§8.2、テーブルはあるが0件なら§8.3を実行する |
| Ingestが `localhost:5432` へ接続して失敗 | Docker内のlocalhostはコンテナ自身 | GCP投入では `cloud-sql-proxy:5432` を指定する |
| Ingestが `[Errno -2] Name or service not known` で全件失敗する | `host.docker.internal` を名前解決できない | `.env.stg`または`.env.prod`を `cloud-sql-proxy:5432` に変更し、§8.1のDocker内Proxy方式で再実行する |
| GCP投入時にローカルPostgreSQLを起動しようとする | GCP用Composeファイルまたは `--no-deps` の指定漏れ | `docker compose -f docker-compose.yml -f docker-compose.gcp.yml run --rm --no-deps ingest` を使用する |
| 別環境のリソースがplanに表示される | workspaceまたはgcloudプロジェクトの選択ミス | applyせず、§5.1と§6.2をやり直す |
| 画面に `Failed to fetch` と表示される | ビルド時のバックエンドURLまたはCORS設定不一致 | フロントを正しい `_BACKEND_URL` で再ビルドし、§9.1で実際のフロントエンドURLを `cors_origins` に設定する。Cloud Runログに `Disallowed CORS origin` がないか確認する |

- `.tfvars`、`.tfstate`、planファイル、ADC、DBパスワードをGitへ登録しない。
- 現在はStaging・ProductionともCloud SQLの削除保護が無効である。本番DBのバックアップ・PITR・削除保護は、公開前に要件に合わせてTerraformへ追加する。
- 現在のCloud SQLはAuth Proxyによるローカル投入のためパブリックIPを有効にしている。より厳格な環境ではプライベートIPとServerless VPC Accessを検討する。
- 現在のDBパスワードはTerraform stateとCloud Run環境変数へ保存される。本格運用ではSecret Manager化を検討する。
- Productionへ直接デプロイせず、Stagingで動作確認後にProductionへ反映する。
