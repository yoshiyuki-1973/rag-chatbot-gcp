# 13 商用環境構築手順書（GCP本番デプロイ - Terraform版）

Terraform (Infrastructure as Code) を使用して、Google Cloud Platform (GCP) 上に検証環境（ステージング）および本番環境（プロダクション）を自動構築する手順。
ローカル開発環境の構築は [12_テスト環境構築手順書.md](12_テスト環境構築手順書.md) を参照。

---

## 1. 前提条件

### 1.1 必要ツール

ローカル環境に以下がインストールされていること。

| ツール | 用途 |
|---|---|
| Google Cloud CLI (`gcloud`) | GCP との認証、Docker 認証 |
| Terraform (v1.0 以上) | GCP インフラのコード管理・構築 |
| Docker | コンテナイメージの検証（ローカル） |

### 1.2 環境の分離（マルチプロジェクト）

セキュリティと安全なリリースのため、GCP プロジェクトを完全に分離します。

- **検証環境 (Staging)**: `rag-chatbot-stg`
- **本番環境 (Production)**: `rag-chatbot-prod`

> [!IMPORTANT]
> コマンドを実行する際は、現在アクティブな GCP プロジェクトが意図した環境（`stg` または `prod`）であるかを必ず確認してください。
> 確認コマンド: `gcloud config get-value project`

---

## 2. デプロイの事前準備

検証・本番それぞれのプロジェクトに対して以下の事前準備を行います。

### 2.1 GCP 認証と初期化

```bash
# GCP アカウントへログイン
gcloud auth login

# 対象のプロジェクトを設定 (例: rag-chatbot-prod)
gcloud config set project [YOUR_PROJECT_ID]

# アプリケーションデフォルト認証情報 (ADC) を初期化 (Terraformが使用)
gcloud auth application-default login
```

### 2.2 Terraform 用環境変数の作成

`terraform/` ディレクトリ内に、環境ごとの変数設定ファイル（`.tfvars`）を作成します。

**例: `terraform/prod.tfvars`**
```hcl
project_id  = "rag-chatbot-prod"
region      = "asia-northeast1"
environment = "prod"
db_password = "YOUR_SECURE_DATABASE_PASSWORD" # ※強力なパスワードを指定
```

> [!IMPORTANT]
> **Cloud SQL のパブリックIPに関するセキュリティ注意点**
> 現在の Terraform テンプレートでは、ローカル PC からの `cloud-sql-proxy` 接続や初期データ投入バッチの実行を簡略化するため、データベースインスタンスにパブリックIPを付与（`ipv4_enabled = true`）しています。
> より厳格な本番運用を行う場合は、セキュリティを高めるためにパブリックIPを無効化（`ipv4_enabled = false`）し、Serverless VPC Access コネクタ等を介したVPC内部のプライベートIP接続に限定する構成へと変更することを強く推奨します。

## 3. インフラ構築手順 (Terraform)

Cloud Run の仕様上、コンテナイメージが Artifact Registry に存在していないとデプロイ時にエラーになります。そのため、**2段階で適用する構成**を取ります。

### Step 1: 共通APIと Artifact Registry / DB の作成

まずはイメージ格納先リポジトリとデータベースを先行して作成します。

```bash
cd terraform

# 初期化
terraform init

# ターゲットを指定して適用 (例: 本番環境)
terraform apply -var-file="prod.tfvars" \
  -target=google_project_service.enabled_apis \
  -target=google_artifact_registry_repository.app_repo \
  -target=google_sql_database_instance.db_instance
```

---

### Step 2: コンテナイメージのビルドとプッシュ

Artifact Registry が作成されたら、Next.js フロントエンドと FastAPI バックエンドのイメージをビルドしてプッシュします。

#### 1. Docker の GCP 認証設定
```bash
gcloud auth configure-docker asia-northeast1-docker.pkg.dev
```

#### 2. バックエンドイメージのビルドとプッシュ
```bash
gcloud builds submit --tag asia-northeast1-docker.pkg.dev/[PROJECT_ID]/rag-chatbot-images-prod/backend:latest ../backend
```

#### 3. フロントエンドイメージのビルドとプッシュ
※ Next.js ビルド時に、バックエンドのエンドポイントを環境変数として静的に埋め込む必要があります。

> [!NOTE]
> **【初回デプロイ時の依存関係の解決手順】**
> 初回デプロイ時は、バックエンドのデプロイが完了するまでバックエンドの正式な URL が確定しません。そのため、以下の手順で循環依存を解決してください。
> 1. 先にバックエンドのみをデプロイ完了させます：
>    `terraform apply -var-file="prod.tfvars" -target=google_cloud_run_v2_service.backend`
> 2. デプロイ完了後、コンソールまたは出力（`terraform output backend_url`）から確定したバックエンドの URL を取得します。
> 3. 取得した URL をビルド引数（`_BACKEND_URL`）に渡して、フロントエンドの Docker イメージをビルド＆プッシュします（下記コマンドを実行）。

# ビルド実行
gcloud builds submit \
  --config=../frontend/cloudbuild.yaml \
  --substitutions=_BACKEND_URL="取得したバックエンドのURL",_PROJECT_ID="[PROJECT_ID]" \
  ../frontend
```

---

### Step 3: 全インフラの適用 (Cloud Run 含む)

イメージがプッシュされたら、残りの Cloud Run および IAM 設定を含めたすべてのリソースを適用します。

```bash
# 全体のデプロイを適用
terraform apply -var-file="prod.tfvars"
```

適用が完了すると、コンソールに以下のような出力が表示されます。

```
Outputs:
artifact_registry_repo = "rag-chatbot-images-prod"
db_connection_name = "rag-chatbot-prod:asia-northeast1:rag-db-prod"
backend_url = "https://rag-chatbot-backend-prod-xxxxx-an.a.run.app"
frontend_url = "https://rag-chatbot-frontend-prod-xxxxx-an.a.run.app"
```

## 4. データベースの初期化とデータインジェスト

データベース構築後、テーブル作成とデータ投入を行います。

### 4.1 テーブルの作成

Cloud SQL Auth Proxy を使用してローカル環境から接続し、テーブル定義を反映します。

1. **Proxy の起動**:
   ```bash
   cloud-sql-proxy [PROJECT_ID]:asia-northeast1:rag-db-prod
   ```
2. **DDL SQL の実行**:
   ローカルの PostgreSQL クライアントから接続し、`docs/05_テーブルレイアウト.md` に記載されている初期化 SQL を実行します。

---

### 4.2 本番データのインジェスト

1. 本番用の環境設定ファイル `.env.prod` を準備します。
   ```bash
   DATABASE_URL=postgresql://app_user:YOUR_DB_PASSWORD@localhost:5432/rag_chatbot
   USE_VERTEX_AI=true
   GCP_PROJECT_ID=[PROJECT_ID]
   GCP_LOCATION=asia-northeast1
   ```
2. データ取り込みバッチを実行します。
   ```bash
   docker compose --env-file .env.prod run --rm ingest
   ```
3. ベクトル検索インデックスを作成します（インジェスト完了後）。
   ```sql
   CREATE INDEX idx_document_chunks_embedding
   ON document_chunks USING ivfflat (embedding vector_cosine_ops)
   WITH (lists = 10);
   ```

## 5. 動作確認チェックリスト

| # | 確認項目 | 方法 |
|---|---|---|
| 1 | バックエンド API が起動している | `GET /health` エンドポイントにアクセスし `status: ok` を確認 |
| 2 | DB 接続が正常 | `/health` の `services.database` が `"ok"` であることを確認 |
| 3 | Vertex AI 接続が正常 | `/health` の `services.llm` が `"ok"` であることを確認 |
| 4 | チャット画面が表示される | `frontend_url` にアクセスして UI が表示されるか確認 |
| 5 | チャットが動作する | 質問を入力し、正常に出典付きで回答が生成されることを確認 |
| 6 | CORS エラーが出ない | ブラウザのデベロッパーツールで CORS エラーの有無を確認 |

---

## 6. Terraform のクリーンアップ (環境の削除)

検証環境などの不要になった GCP リソースを一括削除する場合は、以下のコマンドを実行します。

```bash
# 警告: この操作はDBのデータを含むすべてのリソースを削除します！
terraform destroy -var-file="prod.tfvars"
```
