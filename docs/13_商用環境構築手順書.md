# 13 商用環境構築手順書（GCP本番デプロイ - Terraform版）

Terraform (Infrastructure as Code) を使用して、Google Cloud 上に検証環境（Staging）および本番環境（Production）を構築する手順です。
ローカル開発環境は [12_テスト環境構築手順書.md](12_テスト環境構築手順書.md)、構築後の運用は [15_運用マニュアル.md](15_運用マニュアル.md) を参照してください。

> [!IMPORTANT]
> 本書では環境固有の絶対パスを使用しません。`terraform/`、`backend/`、`frontend/`、`db/` があるディレクトリをプロジェクトルートとします。

---

## 1. 構築する環境

セキュリティ、課金集計、誤操作防止のため、検証環境と本番環境は別の GCP プロジェクトに分離します。

| 環境 | 用途 | プロジェクトIDの例 | Terraform workspace |
|---|---|---|---|
| Staging | デプロイ確認、受入テスト | `your-rag-chatbot-stg` | `stg` |
| Production | 一般公開、本番運用 | `your-rag-chatbot-prod` | `prod` |

プロジェクトIDは全世界で一意であり、作成後は変更できません。表の値は例なので、実際には利用可能なIDを決めてください。

構築される主なリソースは以下です。

- Artifact Registry（Dockerイメージ）
- Cloud SQL for PostgreSQL 15（pgvector）
- Cloud Run（FastAPIバックエンド、Next.jsフロントエンド）
- アプリケーション用サービスアカウント
- Vertex AI / Cloud SQL接続用IAMロール

## 2. 前提条件

### 2.1 必要なアカウント

- Googleアカウント
- 有効な Google Cloud 請求先アカウント
- 対象プロジェクトを作成できる権限
- 請求先アカウントをプロジェクトへ関連付けられる権限

個人のポートフォリオ環境では、プロジェクト作成者がOwnerであれば後続作業を実行できます。組織管理下では、§4.4のIAM権限を管理者へ依頼してください。

### 2.2 必要なローカルツール

| ツール | 確認コマンド | 用途 |
|---|---|---|
| Google Cloud CLI | `gcloud version` | 認証、プロジェクト操作、Cloud Build |
| Terraform 1.0以上 | `terraform version` | GCPリソース構築 |
| Docker Desktop | `docker version` | ローカル確認、DB初期化、Ingest |
| Cloud SQL Auth Proxy v2 | `cloud-sql-proxy --version` | ローカルPCからCloud SQLへ接続 |
| Git | `git --version` | ソースコード管理 |

インストール先は以下です。

- [Google Cloud CLIのインストール](https://cloud.google.com/sdk/docs/install)
- [Terraformのインストール](https://developer.hashicorp.com/terraform/install)
- [Cloud SQL Auth Proxyのインストール](https://cloud.google.com/sql/docs/postgres/connect-auth-proxy#install-proxy)

既にGoogle Cloud CLIを導入済みの場合は、古いバージョンのまま使わないよう更新します。

```powershell
gcloud components update
gcloud version
```

## 3. Google Cloudのアカウント・課金準備

### 3.1 Google Cloudコンソールへログイン

1. [Google Cloudコンソール](https://console.cloud.google.com/) を開く。
2. デプロイに使用するGoogleアカウントでログインする。
3. 初回利用時は利用規約への同意と支払い情報の登録を完了する。

### 3.2 請求先アカウントの作成・確認

1. Google Cloudコンソール左上のナビゲーションメニューを開く。
2. **お支払い（Billing）**を開く。
3. 請求先アカウントがない場合は、画面の案内に従って作成する。
4. 請求先アカウントが有効であることを確認する。

CLIで利用可能な請求先アカウントを確認する場合は次を実行します。

```powershell
gcloud auth login
gcloud billing accounts list --filter="open=true"
```

表示された `ACCOUNT_ID` を後続手順で使用します。請求先アカウントIDや支払い情報はリポジトリへ記録しないでください。

### 3.3 予算アラートの作成（推奨）

想定外の利用を早期に検知するため、プロジェクト作成後に環境ごとの予算アラートを設定します。

1. Google Cloudコンソールで **お支払い** → **予算とアラート**を開く。
2. **予算を作成**を選択する。
3. 対象プロジェクトを指定する。
4. 月額予算と通知しきい値（例: 50%、90%、100%）を設定する。
5. 通知先メールを確認して保存する。

> [!NOTE]
> 予算アラートは通知機能であり、利用料金を自動停止する機能ではありません。

## 4. GCPプロジェクトの作成

### 4.1 プロジェクトIDを決める

次の条件を満たすIDを、StagingとProductionで1つずつ決めます。

- 6～30文字
- 小文字、数字、ハイフンのみ
- 先頭は英字
- 全世界で一意
- 個人名、メールアドレス、秘密情報を含めない

以降は次の変数名で表します。

| 変数 | 内容 |
|---|---|
| `STG_PROJECT_ID` | Stagingの実際のプロジェクトID |
| `PROD_PROJECT_ID` | Productionの実際のプロジェクトID |
| `BILLING_ACCOUNT_ID` | 有効な請求先アカウントID |

### 4.2 Google Cloudコンソールで作成する場合

StagingとProductionについて、それぞれ次を実施します。

1. Google Cloudコンソール上部のプロジェクト選択欄を開く。
2. **新しいプロジェクト**を選択する。
3. プロジェクト名を入力する（例: `RAG Chatbot Staging`）。
4. 一意なプロジェクトIDを入力する。
5. 組織を利用している場合は、所属先の組織またはフォルダを選択する。
6. **作成**を選択する。
7. **お支払い** → **アカウント管理**から、有効な請求先アカウントがリンクされていることを確認する。

### 4.3 gcloudで作成する場合

以下はPowerShellの例です。値を実際のものへ置き換えます。

```powershell
$STG_PROJECT_ID = "your-unique-rag-stg"
$PROD_PROJECT_ID = "your-unique-rag-prod"
$BILLING_ACCOUNT_ID = "000000-000000-000000"

gcloud projects create $STG_PROJECT_ID --name="RAG Chatbot Staging"
gcloud projects create $PROD_PROJECT_ID --name="RAG Chatbot Production"

gcloud billing projects link $STG_PROJECT_ID --billing-account=$BILLING_ACCOUNT_ID
gcloud billing projects link $PROD_PROJECT_ID --billing-account=$BILLING_ACCOUNT_ID
```

組織またはフォルダ配下に作成する場合は、`gcloud projects create`へ `--organization=[ORGANIZATION_ID]` または `--folder=[FOLDER_ID]` を追加します。

作成・課金状態を確認します。

```powershell
gcloud projects describe $STG_PROJECT_ID
gcloud billing projects describe $STG_PROJECT_ID

gcloud projects describe $PROD_PROJECT_ID
gcloud billing projects describe $PROD_PROJECT_ID
```

### 4.4 構築担当者に必要なIAM権限

個人プロジェクトでプロジェクト作成者がOwnerの場合、通常は追加設定不要です。組織管理下で最小権限を付与する場合は、管理者から構築担当者へ次のロールが必要です。

| ロール | ロールID | 用途 |
|---|---|---|
| Service Usage管理者 | `roles/serviceusage.serviceUsageAdmin` | API有効化 |
| Artifact Registry管理者 | `roles/artifactregistry.admin` | Dockerリポジトリ作成 |
| Cloud SQL管理者 | `roles/cloudsql.admin` | DBインスタンス、DB、ユーザー作成 |
| Cloud Run管理者 | `roles/run.admin` | Cloud Runサービス作成 |
| サービスアカウント管理者 | `roles/iam.serviceAccountAdmin` | アプリ用サービスアカウント作成 |
| サービスアカウントユーザー | `roles/iam.serviceAccountUser` | Cloud Runへのサービスアカウント設定 |
| Project IAM管理者 | `roles/resourcemanager.projectIamAdmin` | IAMロール割当 |
| Cloud Build編集者 | `roles/cloudbuild.builds.editor` | ビルド実行 |

権限はStagingとProductionの両プロジェクトへ個別に付与します。サービスアカウントのJSONキーは作成せず、ローカルはADC、Cloud Runはアタッチしたサービスアカウントを使用します。

## 5. gcloudの環境別設定と認証

### 5.1 gcloud configurationを分離する

誤ったプロジェクトへデプロイしないよう、環境ごとにconfigurationを作成します。

```powershell
gcloud config configurations create rag-chatbot-stg
gcloud config set project $STG_PROJECT_ID
gcloud config set run/region asia-northeast1

gcloud config configurations create rag-chatbot-prod
gcloud config set project $PROD_PROJECT_ID
gcloud config set run/region asia-northeast1
```

作業前に対象環境を選択し、必ず確認します。

```powershell
# Stagingを操作する場合
gcloud config configurations activate rag-chatbot-stg

# Productionを操作する場合は prod を指定
# gcloud config configurations activate rag-chatbot-prod

gcloud config get-value project
gcloud auth list
```

### 5.2 Terraform用ADCを作成する

`gcloud auth login`の認証情報とTerraformが使用するApplication Default Credentials（ADC）は別です。

```powershell
gcloud auth login
gcloud auth application-default login
gcloud auth application-default set-quota-project $(gcloud config get-value project)
```

Productionへ切り替えた場合も、`set-quota-project`で現在の対象プロジェクトを設定します。

### 5.3 必要API

Terraformは次のAPIを自動的に有効化します。

- Cloud Run Admin API: `run.googleapis.com`
- Cloud SQL Admin API: `sqladmin.googleapis.com`
- Artifact Registry API: `artifactregistry.googleapis.com`
- Cloud Build API: `cloudbuild.googleapis.com`
- Vertex AI API: `aiplatform.googleapis.com`

手動で先に有効化する場合は、対象configurationを確認してから実行します。

```powershell
gcloud services enable run.googleapis.com sqladmin.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com aiplatform.googleapis.com

gcloud services list --enabled --filter="name:(run.googleapis.com OR sqladmin.googleapis.com OR artifactregistry.googleapis.com OR cloudbuild.googleapis.com OR aiplatform.googleapis.com)"
```

## 6. Terraform環境設定

### 6.1 実行場所

プロジェクトルートから `terraform/` へ移動します。

```powershell
cd terraform
```

以降のTerraformコマンドは `terraform/` で実行します。

### 6.2 環境別workspaceを作成する

StagingとProductionで同じstateを共有すると、別環境のリソースを更新・削除する危険があります。必ずworkspaceを分けます。

```powershell
terraform init

# 初回のみ
terraform workspace new stg
terraform workspace new prod

# 今回構築する環境を選択
terraform workspace select stg
terraform workspace show
```

Productionを構築する場合は `terraform workspace select prod` を使用します。`default` workspaceではデプロイしないでください。

### 6.3 tfvarsを作成する

`terraform/stg.tfvars` と `terraform/prod.tfvars` を作成します。これらは `.gitignore` の `*.tfvars` で除外されます。

`stg.tfvars` の例:

```hcl
project_id   = "your-unique-rag-stg"
region       = "asia-northeast1"
environment  = "stg"
db_password  = "REPLACE_WITH_32_CHAR_URI_SAFE_PASSWORD"
cors_origins = "*"
db_tier      = "db-f1-micro"
```

`prod.tfvars` の例:

```hcl
project_id   = "your-unique-rag-prod"
region       = "asia-northeast1"
environment  = "prod"
db_password  = "REPLACE_WITH_32_CHAR_URI_SAFE_PASSWORD"
cors_origins = "*"
db_tier      = "db-f1-micro"
```

初回はフロントエンドURLが未確定なので `cors_origins = "*"` とし、デプロイ後に§8.2で本番URLへ制限します。

> [!IMPORTANT]
> 現在のTerraformはDBパスワードを接続URLへ直接埋め込みます。URL予約文字による接続エラーを避けるため、十分に長い英大文字・英小文字・数字のみのランダム値を使用してください。tfvars、Terraform state、planファイルをGitへコミットしないでください。

### 6.4 設定を検証する

```powershell
terraform fmt -check
terraform validate
terraform workspace show
gcloud config get-value project
```

`terraform workspace show`とtfvarsの `environment`、`gcloud config get-value project`とtfvarsの `project_id` が一致していることを目視確認します。

## 7. インフラ構築と初回デプロイ

Cloud Runは参照するコンテナイメージがArtifact Registryに存在しないと作成できないため、初回は段階的に適用します。

以下はStagingの例です。Productionではconfiguration、workspace、tfvarsを `prod` に置き換えます。

```powershell
gcloud config configurations activate rag-chatbot-stg
terraform workspace select stg
$TFVARS = "stg.tfvars"
$PROJECT_ID = gcloud config get-value project
$REGION = "asia-northeast1"
$IMAGE_BASE = "${REGION}-docker.pkg.dev/${PROJECT_ID}/rag-chatbot-images"
```

### 7.1 API・Artifact Registry・Cloud SQL・IAMを作成する

```powershell
terraform apply -var-file=$TFVARS -target=google_project_service.enabled_apis -target=google_artifact_registry_repository.app_repo -target=google_sql_database_instance.db_instance -target=google_sql_database.database -target=google_sql_user.db_user -target=google_service_account.app_sa -target=google_project_iam_member.vertex_ai_user -target=google_project_iam_member.cloud_sql_client
```

確認画面で対象プロジェクトと作成リソースを確認し、問題がなければ `yes` を入力します。

> [!NOTE]
> Terraformの `-target` は初回の循環依存を解消するためだけに使用します。通常更新では§9の全体適用を使用してください。

### 7.2 Artifact RegistryへのDocker認証を設定する

```powershell
gcloud auth configure-docker "${REGION}-docker.pkg.dev"
gcloud artifacts repositories describe rag-chatbot-images --location=$REGION
```

### 7.3 バックエンドイメージをビルド・プッシュする

`terraform/`で実行します。

```powershell
gcloud builds submit --tag "${IMAGE_BASE}/backend:latest" ../backend
```

Artifact RegistryのリポジトリIDはTerraform実装に合わせて `rag-chatbot-images` です。

### 7.4 バックエンドを先行デプロイする

```powershell
terraform apply -var-file=$TFVARS -target=google_cloud_run_v2_service.backend -target=google_cloud_run_v2_service_iam_member.backend_public

$BACKEND_URL = terraform output -raw backend_url
$BACKEND_URL
```

URLが表示されたら、Cloud Runコンソールでもバックエンドサービスが作成されていることを確認します。DBスキーマ作成前は `/health` のDB判定がエラーになる場合があります。

### 7.5 フロントエンドイメージをビルド・プッシュする

Next.jsへバックエンドURLをビルド時に埋め込みます。

```powershell
gcloud builds submit --config=../frontend/cloudbuild.yaml --substitutions="_BACKEND_URL=$BACKEND_URL,_PROJECT_ID=$PROJECT_ID" ../frontend
```

### 7.6 全リソースを適用する

```powershell
terraform plan -var-file=$TFVARS -out="${env:COMPUTERNAME}-${TFVARS}.tfplan"
terraform apply "${env:COMPUTERNAME}-${TFVARS}.tfplan"
Remove-Item -LiteralPath "${env:COMPUTERNAME}-${TFVARS}.tfplan"
```

適用後に出力を確認します。

```powershell
terraform output
$FRONTEND_URL = terraform output -raw frontend_url
$BACKEND_URL = terraform output -raw backend_url
```

出力例:

```text
artifact_registry_repo = "projects/your-project/locations/asia-northeast1/repositories/rag-chatbot-images"
db_connection_name     = "your-project:asia-northeast1:rag-db-stg"
backend_url             = "https://rag-chatbot-backend-stg-xxxxx-an.a.run.app"
frontend_url            = "https://rag-chatbot-frontend-stg-xxxxx-an.a.run.app"
```

## 8. Cloud SQLの初期化とRAGデータ投入

### 8.1 Cloud SQL Auth Proxyを起動する

プロジェクトルートへ戻り、接続名を取得します。

```powershell
cd ..\
$CONNECTION_NAME = terraform -chdir=terraform output -raw db_connection_name
cloud-sql-proxy --port 5433 $CONNECTION_NAME
```

Proxyはこのターミナルで起動したままにします。ローカルDBの5432番ポートとの衝突を避けるため、ここでは5433番を使用します。

### 8.2 Dockerのpsqlでスキーマを作成する

別のPowerShellをプロジェクトルートで開きます。`DB_PASSWORD`にはtfvarsと同じ値を設定します。

```powershell
$DB_PASSWORD = "YOUR_DATABASE_PASSWORD"

docker run --rm `
  -e PGPASSWORD=$DB_PASSWORD `
  -v "${PWD}/db:/sql:ro" `
  postgres:15 `
  psql -v ON_ERROR_STOP=1 -h host.docker.internal -p 5433 -U app_user -d rag_chatbot -f /sql/01_schema.sql
```

テーブルを確認します。

```powershell
docker run --rm `
  -e PGPASSWORD=$DB_PASSWORD `
  postgres:15 `
  psql -h host.docker.internal -p 5433 -U app_user -d rag_chatbot -c "\dt"
```

Cloud SQL Studioまたはローカルのpsqlを使う方法は、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) も参照してください。

### 8.3 RAGデータを投入する

投入対象を `data/`、メタ情報を `batch/sources.yaml` に用意し、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) に従って実行します。

Dockerコンテナからホスト上のCloud SQL Auth Proxyへ接続する場合、DBホストは `localhost` ではなく `host.docker.internal`、ポートは `5433` を指定します。

Vertex AIを使用してIngestする場合は、ADCファイルをコンテナへ読み取り専用でマウントします。

```powershell
docker compose run --rm --no-deps `
  -v "${env:APPDATA}/gcloud/application_default_credentials.json:/tmp/gcp-adc.json:ro" `
  -e GOOGLE_APPLICATION_CREDENTIALS=/tmp/gcp-adc.json `
  -e DATABASE_URL="postgresql://app_user:${DB_PASSWORD}@host.docker.internal:5433/rag_chatbot" `
  -e USE_VERTEX_AI=true `
  -e GCP_PROJECT_ID=$PROJECT_ID `
  -e GCP_LOCATION=$REGION `
  ingest
```

投入後、必要に応じてRAGデータ投入マニュアルのベクトル検索インデックス再構築・確認手順を実行します。

### 8.4 Proxyを終了する

データベース作業が完了したら、Cloud SQL Auth Proxyを起動したターミナルで `Ctrl+C` を押します。

## 9. CORS制限と最終適用

### 9.1 フロントエンドURLをtfvarsへ設定する

初回デプロイ後、対象環境のtfvarsを編集します。

```hcl

```

ProductionではProductionのフロントエンドURLを指定します。末尾の `/` は付けません。

### 9.2 全体を再適用する

`terraform/`で、対象workspaceを確認して実行します。

```powershell
terraform workspace show
gcloud config get-value project
terraform plan -var-file=$TFVARS
terraform apply -var-file=$TFVARS
```

## 10. 動作確認

```powershell
$BACKEND_URL = terraform output -raw backend_url
$FRONTEND_URL = terraform output -raw frontend_url

Invoke-RestMethod "$BACKEND_URL/health"
Start-Process $FRONTEND_URL
```

| # | 確認項目 | 合格条件 |
|---|---|---|
| 1 | バックエンドAPI | `/health` の `status` が `ok` |
| 2 | Cloud SQL接続 | `/health` の `services.database` が `ok` |
| 3 | Vertex AI接続 | `/health` の `services.llm` が `ok` |
| 4 | フロントエンド | Cloud Run URLでチャット画面が表示される |
| 5 | RAG回答 | 投入データへの質問で回答と出典が表示される |
| 6 | CORS | ブラウザの開発者ツールにCORSエラーがない |
| 7 | 環境分離 | StagingとProductionでURL、DB、プロジェクトが別である |

Cloud Runを一般公開できない場合は、組織ポリシーで `allUsers` が禁止されている可能性があります。組織管理者へ公開方針を確認してください。

## 12. 環境の削除

> [!CAUTION]
> `terraform destroy` はCloud SQLのデータを含む対象workspaceのリソースを削除します。実行前にバックアップと対象環境を確認してください。現在のTerraformはCloud SQLの削除保護を無効にしています。

```powershell
gcloud config get-value project
terraform workspace show
terraform plan -destroy -var-file=$TFVARS
terraform destroy -var-file=$TFVARS
```

Stagingだけを削除する場合は、必ずStagingのgcloud configurationとTerraform workspaceを選択します。

## 13. トラブルシューティング

| 症状 | 主な原因 | 対処 |
|---|---|---|
| `terraform` が見つからない | Terraform未インストール、PATH未設定 | §2.2の公式手順でインストール後、ターミナルを開き直す |
| API有効化が403 | Service Usage管理者権限不足、課金未設定 | IAMとBillingリンクを確認する |
| Cloud BuildがArtifact Registryへpushできない | Cloud Build実行サービスアカウントの権限不足 | 同一プロジェクトか確認し、必要に応じてArtifact Registry Writerを付与する |
| Cloud Runで `allUsers` を設定できない | 組織ポリシーで公開アクセス禁止 | 組織管理者へ確認し、認証付き公開へ変更する |
| `/health` が `database: error` | スキーマ未作成、DB接続情報不一致、Cloud SQL権限不足 | §8の初期化、Cloud SQL Clientロール、Cloud Runログを確認する |
| `/health` が `llm: error` | Vertex AI API無効、リージョン・IAM不一致 | API、`GCP_PROJECT_ID`、`GCP_LOCATION`、Vertex AI Userロールを確認する |
| Ingestが `localhost:5432` へ接続して失敗 | Docker内のlocalhostはコンテナ自身 | `host.docker.internal:5433` を指定する |
| 別環境のリソースがplanに表示される | workspaceまたはgcloudプロジェクトの選択ミス | applyせず、§5.1と§6.2をやり直す |
| フロントからAPIへ接続できない | ビルド時URLまたはCORS設定不一致 | フロントを正しい `_BACKEND_URL` で再ビルドし、tfvarsの `cors_origins` を更新する |

## 14. セキュリティ上の注意

- サービスアカウントのJSONキーは作成しない。
- `.tfvars`、`.tfstate`、planファイル、ADC、DBパスワードをGitへ登録しない。
- 本番DBのバックアップ・PITR・削除保護は、公開前に要件に合わせてTerraformへ追加する。
- 現在のCloud SQLはAuth Proxyによるローカル投入のためパブリックIPを有効にしている。より厳格な環境ではプライベートIPとServerless VPC Accessを検討する。
- 現在のDBパスワードはTerraform stateとCloud Run環境変数へ保存される。本格運用ではSecret Manager化を検討する。
- Productionへ直接デプロイせず、Stagingで動作確認後にProductionへ反映する。
