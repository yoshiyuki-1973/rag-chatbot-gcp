# 01 要件定義

## 1. システム概要

### 1.1 システムの目的

スポーツに関する公開データ（Markdown・PDF）を取り込み、ユーザーの質問に対して **出典付きで回答する RAG（Retrieval-Augmented Generation）チャットボット** を構築する。

本プロジェクトは以下を目的として開発するポートフォリオ作品である。

- RAG システムの設計力・実装力を示す
- ドキュメント取り込み（Ingest）→ 検索 → 生成という一連の流れを実装する
- 低コスト（無料枠中心）で動く実用的なシステムを構築する
- LLM・ベクトルストアを差し替え可能な設計にすることで、拡張性を示す

### 1.2 ユースケース

| # | ユースケース | アクター | 概要 |
|---|---|---|---|
| UC-01 | チャット質問 | エンドユーザー | スポーツに関する質問を入力し、出典付きの回答を得る |
| UC-02 | 出典確認 | エンドユーザー | 回答に付随する出典（ドキュメント名・URL）を確認する |
| UC-03 | 検索 | エンドユーザー | キーワードでドキュメントを検索し、関連チャンクを取得する |
| UC-04 | データ取り込み | システム管理者 | Markdown・PDF ファイルをバッチ処理でベクトル DB に登録する |
| UC-05 | ヘルスチェック | 監視システム | API サーバーの死活監視を行う |

### 1.3 想定ユーザー

- **エンドユーザー**: スポーツに関心があり、信頼できる情報源に基づく回答を求める一般ユーザー
- **ポートフォリオ閲覧者**: RAG システムの実装例を評価する採用担当者・エンジニア
- **システム管理者**: データ投入・運用を行う開発者自身

---

## 2. 機能要件

### 2.1 チャット機能

- ユーザーが自然言語で質問を入力できること
- 回答に使用したドキュメントの出典（タイトル・URL・信頼性スコア）を表示すること
- 会話履歴を保持し、文脈を踏まえた回答ができること（ただし MVP ではオプション）
- 回答生成が完了するまでローディング状態を表示すること

### 2.2 検索機能

- クエリのベクトル類似検索（pgvector）で関連チャンクを取得できること
- 取得チャンク数（top-k）を設定できること

### 2.3 データ取り込み機能

- Markdown ファイルを取り込み、チャンク分割・embedding 生成・DB 登録できること
- PDF ファイルを取り込み、テキスト抽出・チャンク分割・embedding 生成・DB 登録できること
- 既登録のドキュメントを再取り込みする場合は冪等性を担保すること（source_url で重複排除）

---

## 3. 非機能要件

| 項目 | 要件 |
|---|---|
| コスト | 無料枠（Vercel Hobby・Render Free・Supabase Free）で運用できること |
| 応答速度 | チャット回答 10 秒以内（LLM レイテンシに依存） |
| 可用性 | ポートフォリオ用途のため SLA は設定しない（Render Free はスリープあり） |
| セキュリティ | API キーを環境変数で管理、コードにハードコードしない |
| 拡張性 | LLM・ベクトルストアを差し替え可能な抽象層を設ける |
| 保守性 | テストコードを整備し、モック戦略を明確にする |

---

## 4. 制約条件

- Gemini（Google）の API を LLM として使用する（将来 Claude・OpenAI に差し替え可能）
- ベクトル DB は Supabase（pgvector）を使用する（将来 Pinecone・Chroma に差し替え可能）
- バッチ処理はローカル Docker で実行し、クラウドジョブスケジューラは使わない
- フロントエンドは Next.js、バックエンドは FastAPI とする
- データソースはスポーツに関する公開 Markdown・PDF に限定する
