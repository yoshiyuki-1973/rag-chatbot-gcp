## 2. 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **Next.js（Vercel）** | チャット UI の提供。ユーザーの質問入力・回答表示・出典表示を担当。SSR/CSR を適切に使い分ける。 |
| **FastAPI（Render）** | RAG の中核ロジックを担う REST API。検索・LLM 呼び出し・レスポンス生成を行う。 |
| **RAG Service** | クエリの embedding 生成 → ベクトル検索 → コンテキスト組み立て → LLM 呼び出し → 回答生成の一連を制御する。 |
| **LLM Adapter** | LLM の差し替えを可能にする抽象層。現在は Gemini を実装。インターフェースを満たせば Claude・OpenAI に切り替えられる。 |
| **Vector Store Adapter** | ベクトル検索の差し替えを可能にする抽象層。現在は Supabase pgvector を実装。 |
| **Supabase（PostgreSQL + pgvector）** | ドキュメントメタ情報・チャンクテキスト・ベクトル・会話履歴を保存する。pgvector 拡張で高速近傍検索を行う。 |
| **Gemini API（Google AI Studio）** | テキスト生成 LLM。RAG で取得したコンテキストを元に回答を生成する。 |
| **Ingest Pipeline（ローカル Docker）** | Markdown・PDF をテキスト化し、チャンク分割・embedding 生成・DB 登録を行うバッチ処理。 |

## 3. データフロー

### 3.1 チャット（クエリ → 回答）フロー

```
1. ユーザーが質問を入力（Next.js）
2. POST /chat にクエリを送信（FastAPI）
3. クエリの embedding を生成（Embedding モデル）
4. pgvector で類似チャンクを top-k 件取得（Supabase）
5. チャンクテキストとメタ情報からコンテキストを組み立てる
6. システムプロンプト + コンテキスト + ユーザークエリを LLM に送信（Gemini API）
7. LLM から回答テキストを取得
8. 回答テキスト + 出典リスト（document_id・title・source_url・authority_score）を返却
9. Next.js で回答と出典を表示
```

### 3.2 データ取り込み（Ingest）フロー

```
1. ローカルの data/ ディレクトリに Markdown / PDF を配置
2. ingest.py を Docker コンテナ内で実行
3. ファイルタイプを判定し、対応するパーサーでテキスト抽出
4. 独自 chunker でチャンク分割（chunk_size=500, overlap=50）
5. embedding モデルで各チャンクのベクトルを生成
6. documents テーブルにドキュメントメタ情報を upsert（source_url で重複排除）
7. document_chunks テーブルに各チャンクを insert
8. 処理完了ログを出力
```

## 4. 技術スタックと採用理由

### 4.1 Supabase（PostgreSQL + pgvector）を採用した理由

- **無料枠が充実**: Supabase Free Tier は PostgreSQL 500MB、API アクセス、認証機能を無料で提供する。
- **pgvector で SQL のままベクトル検索が可能**: 専用ベクトル DB（Pinecone 等）を追加せずに、RDB とベクトル検索を 1 つのサービスで完結できる。これによりインフラの複雑さを下げコストを最小化できる。
- **将来の差し替えに対応**: Vector Store Adapter を設けることで、将来 Pinecone・Chroma・Weaviate への切り替えが容易になる。
- **メタ情報との結合が容易**: documents テーブルと document_chunks テーブルを JOIN することで、チャンク検索結果に出典メタ情報を付与できる。

### 4.2 FastAPI を採用した理由

- **Python エコシステムとの親和性**: LangChain・OpenAI SDK・PyMuPDF など RAG で必要になるライブラリはほぼ Python 製であり、同じ言語で API を書ける。
- **高速・非同期対応**: `async/await` ネイティブ対応で、LLM API 呼び出し等の I/O バウンド処理を効率化できる。
- **自動ドキュメント生成**: `/docs`（Swagger UI）が自動生成され、API 仕様書の作成コストを削減できる。
- **型ヒントと Pydantic**: リクエスト・レスポンスの型定義が明確になり、バグを減らせる。

### 4.3 Next.js を採用した理由

- **Vercel との相性**: 公式サポートで CI/CD が最も簡単に構築できる。
- **SSR + CSR の使い分け**: 初期ページは SSR でレンダリング、チャット部分は CSR でリアルタイム更新できる。
- **App Router（RSC）**: Server Components でフロントの不要な JS を削減しパフォーマンスを改善できる。
- **TypeScript 標準**: 型安全な開発が可能。

### 4.4 Render / Vercel を採用した理由

- **Render Free Tier**: FastAPI（Python）を無料でホスティングできる。GitHub 連携で自動デプロイ。スリープがあるが、ポートフォリオ用途では許容できる。
- **Vercel Hobby Tier**: Next.js を無料でホスティング・CDN 配信できる。デプロイが非常に簡単。
- いずれも GitHub との連携で CI/CD が自動化され、運用コストが低い。

### 4.5 Gemini（Google）を採用した理由

- **無料枠・低コスト**: ポートフォリオ開発時点で低コスト利用が可能。
- **OpenAI 互換 API**: OpenAI SDK の `base_url` を変更するだけで接続できるため、将来 Claude・OpenAI・その他モデルへの切り替えが最小コストで行える。
- LLM Adapter の抽象層を設けることで、Gemini の位置づけは「現在の実装」に過ぎず、ビジネスロジックへの影響なく差し替えられる。

### 4.6 ローカル Docker でバッチ処理を行う理由

- **コスト削減**: クラウドのジョブスケジューラ（Cloud Run Jobs 等）は起動コストがかかる。データ取り込みは開発時・更新時のみ実行するため、ローカル実行で十分。
- **開発効率**: ローカルでファイルを配置してすぐ実行でき、デバッグが容易。
- **環境の一致**: Docker により「動いた環境」をそのまま再現でき、依存関係の問題を防ぐ。
