## 【オープニング】0:00〜

こんにちは。  
今回は、私が作成したポートフォリオ作品「スポーツルール RAG チャットボット」を紹介します。

RAG とは Retrieval-Augmented Generation の略で、  
「検索して取得した情報を根拠に LLM に回答させる」仕組みのことです。  
このアプリでは、スポーツに関する公開資料を取り込み、  
ユーザーの質問に対して **出典付きで回答** を返します。

---

## 【このアプリで何ができるか】0:30〜

たとえば「サッカーのオフサイドルールを教えてください」と質問すると、  
回答テキストだけでなく、  
「FIFA サッカー競技規則 2024/25」「JFA サッカー解説」といった  
参照元のドキュメント名・URL・信頼性スコアも一緒に表示されます。

これにより、ユーザーは「どの資料に基づいて答えているか」を確認できます。  
特にルールや制度の説明では、根拠が見えることが信頼性につながります。

---

## 【技術スタック】1:15〜

全体の技術構成を説明します。

| レイヤー | 技術 |
|---|---|
| フロントエンド | Next.js 14 + TypeScript |
| バックエンド | FastAPI（Python 3.12） |
| DB・ベクトル検索 | Supabase（PostgreSQL + pgvector） |
| Embedding | OpenAI text-embedding-3-small |
| LLM | Gemini（Google） |
| バッチ処理 | Docker Compose で実行する Ingest Pipeline |
| ホスティング | フロント：Vercel ／ バック：Render |

この構成を選んだ理由は 3 つあります。

**1 つ目は、役割の分離です。**  
UI・API・データ処理を分離し、それぞれの責務を明確にしています。

**2 つ目は、検索品質の担保です。**  
pgvector でコサイン類似度検索を行い、  
類似度スコアやソース情報をレスポンスに含めています。

**3 つ目は、将来の拡張性です。**  
LLM は抽象化レイヤーを設けており、  
Gemini から Claude や OpenAI へ差し替え可能な設計にしています。

---

## 【質問から回答までの処理フロー】2:30〜

ユーザーが質問を送信してから回答が返るまでの流れを説明します。

```
① ユーザーが質問を入力して送信
        ↓
② フロントエンド（Next.js）が POST /chat を呼び出す
        ↓
③ バックエンド（FastAPI）が質問文を受け取る
        ↓
④ OpenAI Embeddings API で質問文をベクトルに変換
        ↓
⑤ pgvector でコサイン類似度検索（上位 5 チャンク取得）
        ↓
⑥ 取得チャンクをコンテキストとして整形
        ↓
⑦ Gemini にシステムプロンプト＋コンテキスト＋質問を渡す
        ↓
⑧ 回答テキスト＋出典情報をフロントへ返却
        ↓
⑨ 画面に回答と出典バッジを表示
```

LLM に渡すシステムプロンプトには、  
「コンテキスト外の情報を使わないこと」  
「情報が無ければ無いと答えること」  
「推測や創作はしないこと」  
「日本語で回答すること」  
の 4 つのルールを明示しています。  
これにより、ハルシネーション（でたらめな回答）を抑制しています。

## 【コードの設計ポイント】4:00〜

### LLM の抽象化

`BaseLLMClient` という抽象基底クラスを定義し、  
具体実装（`GeminiLLMClient` 等）をその下に置いています。  
`main.py` の `_create_llm_client()` ファクトリが環境変数 `LLM_PROVIDER` を見てクライアントを生成します。  
新しい LLM を追加するときは `elif` を 1 行と Adapter クラスを追加するだけで済みます。

### 依存性注入

FastAPI の `Depends` を使い、  
`RAGService` に `EmbeddingClient`・`SupabaseVectorStore`・`BaseLLMClient` を注入しています。  
テスト時にはモックに差し替えられるため、外部 API に依存しない単体テストが可能です。

### エラーハンドリング

検索失敗と LLM 失敗を分けて捕捉し、  
ユーザー向けには日本語メッセージを返します。  
スタックトレースはサーバーログにのみ出力し、レスポンスには含めません。

---

## 【データ取り込みバッチ】5:15〜

スポーツルール資料を検索できるようにするには、  
事前にデータを取り込む必要があります。  
その処理を行うのが Ingest Pipeline です。

```
data/ ディレクトリに Markdown・PDF を置く
        ↓
sources.yaml にメタ情報（タイトル・URL・組織・信頼性スコア）を記載
        ↓
docker compose run --rm ingest を実行
        ↓
① テキスト抽出（Markdown / PyMuPDF）
② チャンク分割（500 トークン・50 トークンオーバーラップ）
③ Embedding 生成（OpenAI API へ 100 件ずつ送信）
④ Supabase に documents・document_chunks を upsert
```

チャンクに 50 トークンのオーバーラップを持たせているのは、  
チャンク境界で文脈が途切れて検索精度が下がるのを防ぐためです。

## 【画面と API】6:15〜

画面は 2 つあります。

**チャット画面（/）**  
質問を入力して送信すると、  
ユーザー発話とアシスタント回答が時系列で並びます。  
回答の下に出典バッジが表示され、ドキュメント名・組織名・信頼性スコアを確認できます。

**検索画面（/search）**  
LLM を使わずにベクトル検索だけを実行して、  
関連チャンクの内容と類似度を直接確認できます。  
RAG の検索精度をデバッグする用途に使えます。

API は 3 本です。  
`POST /chat`（RAG 回答）、`POST /search`（検索のみ）、`GET /health`（死活監視）。  
`/health` では DB と LLM の接続状態をそれぞれ返すため、  
障害時の切り分けがしやすくなっています。

## 【テスト】7:15〜

バックエンドでは pytest を用いてテストを用意しています。

- **単体テスト**：設定クラス・LLM クライアント生成ファクトリ・RAGService のモックテスト  
- **統合テスト**：ヘルスチェック API・依存サービス異常系の挙動確認

外部 API（OpenAI・Gemini・Supabase）は `unittest.mock` でモック化し、  
ネットワーク接続なしでテストが実行できるようにしています。

---

## 【まとめと今後の展望】8:00〜

このプロジェクトで意識したのは、  
「動けばいい」ではなく、  
差し替えやすく・テストしやすく・運用しやすい設計にすることです。

今後の改善候補は 3 つあります。

**1 つ目：再ランキングの導入**  
ベクトル検索後に Cross-Encoder で再スコアリングし、検索精度をさらに高める。

**2 つ目：認証・レート制限の追加**  
JWT 認証と slowapi によるレート制限を追加し、公開運用に対応する。

**3 つ目：マルチターン対応**  
会話履歴を活用して、複数ターンにわたって文脈を保持した応答を実現する。
