# 12 テスト環境構築手順書（ローカル開発）

ローカル PC 上で開発・テストを行うための環境構築手順。
本番環境の構築は [13_商用環境構築手順書.md](13_商用環境構築手順書.md) を参照すること。

---

## 1. 前提条件

### 1.1 必要ツール

| ツール | バージョン | 確認コマンド | 用途 |
|---|---|---|---|
| Docker Desktop | 最新版 | `docker --version` | バックエンド・バッチの実行 |
| Node.js | 20.x 以上 | `node --version` | フロントエンド開発 |
| Python | 3.12 以上 | `python --version` | テスト直接実行時 |
| Git | 最新版 | `git --version` | ソースコード管理 |

### 1.2 必要アカウント・APIキー

テスト環境では以下を用意する。**本番用の Supabase プロジェクトとは必ず分けること。**

| サービス | 用途 | 注意 |
|---|---|---|
| Supabase（Free Tier） | ローカル開発用 DB | 本番プロジェクトとは別に作成する |
| Google AI Studio Gemini API | LLM 呼び出し | 開発中も実 API を使用する（無料枠を使用） |
| OpenAI API | Embedding 生成 | 開発中も実 API を使用する |

#### Google AI Studio（Gemini API）

1. [aistudio.google.com](https://aistudio.google.com) にサインインする
2. 左メニューの「API Keys」→「Create API Key」をクリック
3. Key name を入力（例: `rag-chatbot-dev`）し「Create」
4. 表示された `your-gemini-api-key...` の文字列をコピーし、安全な場所に保存する

> **このキーは作成直後にしか確認できない。必ずコピーすること。**

#### OpenAI（Embedding API）

1. [platform.openai.com](https://platform.openai.com) にサインインする
2. 右上のアカウントアイコン →「API keys」を選択
3. 「Create new secret key」をクリック
4. Name を入力（例: `rag-chatbot-dev`）し「Create secret key」
5. 表示された `sk-xxx...` の文字列をコピーし、安全な場所に保存する

> **このキーは作成直後にしか確認できない。必ずコピーすること。**

> テスト（pytest）はすべてモックを使用するため、テスト実行時に API キーは不要。

## 3. テスト用 Supabase のセットアップ

### 3.1 プロジェクト作成

1. [supabase.com](https://supabase.com) にサインインし「New Project」を作成する
2. プロジェクト名を `rag-chatbot-dev`（本番は `rag-chatbot-prod`）にするなど区別できる名前にする

### 3.2 スキーマの作成

SQL Editor を開き、以下を順番に実行する。

```sql
-- pgvector 拡張の有効化
CREATE EXTENSION IF NOT EXISTS vector;

-- documents テーブル
CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    source_url      TEXT NOT NULL UNIQUE,
    file_type       TEXT NOT NULL CHECK (file_type IN ('markdown', 'pdf')),
    organization    TEXT,
    authority_score NUMERIC(3, 2) NOT NULL DEFAULT 0.50 CHECK (authority_score BETWEEN 0 AND 1),
    content_date    DATE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- document_chunks テーブル
CREATE TABLE document_chunks (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id   UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index   INTEGER NOT NULL,
    content       TEXT NOT NULL,
    embedding     VECTOR(1536),
    token_count   INTEGER,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);

-- chat_history テーブル
CREATE TABLE chat_history (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id  TEXT NOT NULL,
    query       TEXT NOT NULL,
    response    TEXT NOT NULL,
    sources     JSONB NOT NULL DEFAULT '[]',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- updated_at 自動更新トリガー
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_documents_updated_at
BEFORE UPDATE ON documents
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- インデックス
CREATE INDEX idx_documents_organization ON documents (organization);
CREATE INDEX idx_chat_history_session_id ON chat_history (session_id);
CREATE INDEX idx_chat_history_created_at ON chat_history (created_at);

-- ベクトル検索インデックス（全チャンク insert 後に作成すること）
-- lists は sqrt(行数) 程度が推奨。初期は 10 で十分、1万件超で 100 に上げる
CREATE INDEX idx_document_chunks_embedding
ON document_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 10);
```

### 3.3 接続文字列の取得

Supabase ダッシュボード → Settings → Database → **Connection string (URI)** からコピーする。次の §4 で `.env` に設定する。

## 4. 環境変数の設定

```bash
cp .env.example .env
```

`.env` を編集する。**テスト環境専用の値を設定すること。**

```bash
# ── テスト環境用 Supabase ──────────────────────────────────────────
# §3.3 で取得した接続文字列を設定する
DATABASE_URL=postgresql://postgres:[password]@[host]:5432/postgres

# ── LLM / Embedding ───────────────────────────────────────────────
# §1.2 で取得した Google AI Studio Gemini API キー
GEMINI_API_KEY=your-gemini-api-key

# §1.2 で取得した OpenAI API キー
OPENAI_API_KEY=sk-xxx

# LLM プロバイダー（MVP は gemini のみ対応）
LLM_PROVIDER=gemini

# ── ローカル開発用 ─────────────────────────────────────────────────
# バックエンド起動後にフロントエンドから接続する URL
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000

# デフォルト検索件数（1〜20）
TOP_K_DEFAULT=5

# CORS 許可オリジン（ローカル開発はそのまま）
CORS_ORIGINS=http://localhost:3000
```

## 5. バックエンド（FastAPI）の起動

### 5.1 Docker を使う場合（推奨）

```bash
docker compose up backend
```

### 5.2 直接起動する場合

```bash
cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 5.3 動作確認

```bash
curl http://localhost:8000/health
# 期待レスポンス: {"status":"ok","version":"0.1.0","services":{"database":"ok","llm":"ok"}}
```

Swagger UI: `http://localhost:8000/docs`

## 6. フロントエンド（Next.js）の起動

```bash
cd frontend
npm install
npm run dev
```

確認: `http://localhost:3000` にアクセスしてチャット画面が表示されれば成功。

## 7. テストの実行

```bash
cd backend

# 全テスト実行
py -3.13 -m pytest tests/ -v

# カバレッジ付き
py -3.13 -m pytest tests/ -v --cov=app --cov-report=term-missing
```

> テストはすべてモックを使用するため、`.env` の設定や Supabase への接続は不要。

### テスト構成

| ディレクトリ | 対象 | 実行時間目安 |
|---|---|---|
| `tests/unit/` | RAGService・LLM ファクトリ・Settings | 数秒 |
| `tests/integration/` | `/health` エンドポイント | 数秒 |

## 8. Ingest バッチの実行（テストデータ投入）

1. `data/` ディレクトリにファイルを配置する

```
data/
├── markdown/
│   └── sample.md
└── pdf/
    └── sample.pdf
```

2. `batch/sources.yaml` にメタ情報を記述する（[06_RAG設計.md §3.2](06_RAG設計.md) 参照）

3. バッチを実行する

```bash
docker compose run --rm ingest
```

## 9. docker-compose.yml の構成

```yaml
services:
  backend:
    build:
      context: ./backend
    ports:
      - "8000:8000"      # ホスト:コンテナ
    env_file: .env
    volumes:
      - ./backend:/app   # ホットリロード用マウント
    command: uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

  ingest:
    build:
      context: ./batch
    env_file: .env
    volumes:
      - ./data:/app/data:ro        # データファイルを読み取り専用でマウント
      - ./batch/sources.yaml:/app/sources.yaml:ro
    profiles:
      - batch                      # docker compose run --rm ingest で単発実行
```

> フロントエンドは `npm run dev` でローカル起動するため docker-compose には含めない。

## 10. トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| `/health` で `database: error` | `DATABASE_URL` が正しくない / Supabase が停止している | `.env` を確認・Supabase ダッシュボードで再起動 |
| `/health` で `llm: error` | `GEMINI_API_KEY` が正しくない / Google AI Studio API が不安定 | キーを確認・数分後に再試行 |
| `docker compose up backend` でポート衝突 | 8000 番ポートが使用中 | `lsof -i :8000` で確認し既存プロセスを停止 |
| バッチで `embedding` エラー | `OPENAI_API_KEY` が正しくない / レート制限 | キーを確認・レート制限の場合は数分待機 |
| `pytest` で `ImportError` | Python バージョンが古い | Python 3.12 以上を使用する |
