# 12 テスト環境構築手順書（ローカル開発）

ローカル PC 上で開発・テストを行うための環境構築手順。
本番・検証環境の構築は [13_商用環境構築手順書.md](13_商用環境構築手順書.md) を参照。

---

## 1. 前提条件

### 1.1 必要ツール

| ツール | 用ンテージ | 確認コマンド | 用途 |
|---|---|---|---|
| Docker Desktop | 最新版 | `docker --version` | バックエンド・データベース・バッチの実行 |
| Node.js | 20.x 以上 | `node --version` | フロントエンド開発 |
| Python | 3.12 以上 | `python --version` | テスト直接実行時 |
| Git | 最新版 | `git --version` | ソースコード管理 |

### 1.2 必要アカウント・APIキー

ローカル開発環境では以下を用意します。

| サービス | 用途 | 注意 |
|---|---|---|
| Google AI Studio Gemini API | LLM 呼び出し / Embedding 生成 | 開発中も実 API を使用する（無料枠を使用） |

#### Google AI Studio（Gemini API）

1. [aistudio.google.com](https://aistudio.google.com) にサインインする
2. 左メニューの「API Keys」→「Create API Key」をクリック
3. Key name を入力（例: `rag-chatbot-gcp-dev`）し「Create」
4. 表示された API キーをコピーし、安全な場所に保存する

> [!NOTE]
> バックエンドのユニットテスト（pytest）はすべてモックを使用するため、テスト実行時に API キーは不要です。

## 2. リポジトリのクローン

```bash
git clone https://github.com/<username>/rag-chatbot-gcp.git
cd rag-chatbot-gcp
```

以降の `docker compose` コマンドは、`docker-compose.yml` があるプロジェクトルートで実行します。

プロジェクトの保存場所は環境ごとに異なります。本書では絶対パスを固定せず、`docker-compose.yml` があるディレクトリをプロジェクトルートとして扱います。現在位置は次のコマンドで確認できます。

```powershell
git rev-parse --show-toplevel
Test-Path .\docker-compose.yml  # True ならプロジェクトルート
```

## 3. Docker Compose 設定ファイルの使い分け

このプロジェクトでは、Docker Compose の設定を共通設定とローカル開発設定に分けています。

| ファイル | 用途 | 内容 |
|---|---|---|
| `docker-compose.yml` | 共通設定 | サービス定義、ビルド設定、環境変数ファイル、DB ボリュームなど |
| `docker-compose.override.yml` | ローカル開発用 | ポート公開、ソースコードのマウント、ホットリロード、初期化 SQL / データ投入ファイルのマウント |

通常のローカル開発では、以下のように `docker compose` コマンドを実行すれば `docker-compose.override.yml` が自動で読み込まれます。

```bash
docker compose up backend
```

ローカル開発時に有効になる主なマウントは以下です。

| サービス | マウント | 目的 |
|---|---|---|
| `backend` | `./backend:/app` | Python コード変更をコンテナに即反映し、`uvicorn --reload` で再読み込みする |
| `db` | `./db/01_schema.sql:/docker-entrypoint-initdb.d/01_schema.sql:ro` | DB ボリューム初回作成時にテーブルスキーマを自動作成する |
| `ingest` | `./data:/app/data:ro` | ローカルの投入データをバッチコンテナから読み込む |
| `ingest` | `./batch/sources.yaml:/app/sources.yaml:ro` | ローカルのデータソース定義をバッチコンテナから読み込む |

> [!NOTE]
> 本番の Cloud Run / Cloud SQL 構成は Terraform と Dockerfile を使ってデプロイします。ローカル開発用の `docker-compose.override.yml` は本番デプロイには使用しません。

## 4. Docker イメージのビルド

バックエンドと Ingest バッチの Docker イメージをビルドします。

```bash
# プロジェクトルートで実行
docker compose build
```

個別にビルドする場合は、サービス名を指定します。

```bash
docker compose build backend
docker compose build ingest
```

> [!NOTE]
> `docker compose up` 実行時にも必要に応じてビルドされますが、初回セットアップでは先に `docker compose build` を実行しておくと、依存関係や Dockerfile の問題を切り分けやすくなります。

## 5. テスト用ローカルデータベースのセットアップ

ローカル開発では Docker を使用して PostgreSQL (pgvector 拡張付き) を起動します。

### 5.1 データベースの起動

```bash
# プロジェクトルートで実行
# データベースコンテナのみを起動
docker compose up -d db
```

### 5.2 テーブルスキーマの作成

テーブルスキーマは `db/01_schema.sql` に定義しています。
DB ボリュームが空の状態で `db` コンテナを初回起動すると、Docker の初期化処理により自動で作成されます。

既存の DB コンテナに対して手動で反映する場合は、以下のコマンドを実行します。

```bash
# プロジェクトルートで実行
docker compose exec db psql -v ON_ERROR_STOP=1 -U postgres -d rag_chatbot -f /docker-entrypoint-initdb.d/01_schema.sql
```

作成済みテーブルを確認する場合は、以下を実行します。

```bash
# プロジェクトルートで実行
docker compose exec db psql -U postgres -d rag_chatbot -c "\dt"
```

> [!NOTE]
> `db/01_schema.sql` は再実行してもエラーになりにくいように、`CREATE TABLE IF NOT EXISTS` と `CREATE INDEX IF NOT EXISTS` を使用しています。

## 6. 環境変数の設定

```bash
cp .env.example .env
```

`.env` を編集し、ローカル開発用の値を設定します。

```bash
# ── ローカルデータベース ──────────────────────────────────────────
# ホストOSから接続する場合は localhost を指定
# (コンテナ内から接続する場合は docker-compose によって自動的に `db` が名前解決されます)
DATABASE_URL=postgresql://postgres:postgres_password@db:5432/rag_chatbot

# ── LLM / Embedding (Google AI Studio) ─────────────────────────
GEMINI_API_KEY=your-gemini-api-key
GEMINI_MODEL=gemini-2.5-flash
GEMINI_FALLBACK_MODELS=gemini-2.5-flash-lite

# ── GCP / Vertex AI 設定 (ローカル開発では通常 false) ────────────────
USE_VERTEX_AI=false
GCP_PROJECT_ID=
GCP_LOCATION=us-central1

# ── アプリケーション設定 ─────────────────────────────────────────
LLM_PROVIDER=gemini
TOP_K_DEFAULT=10
CORS_ORIGINS=http://localhost:3000
DB_POOL_MIN_SIZE=1
DB_POOL_MAX_SIZE=3
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

## 7. バックエンド（FastAPI）の起動

### 7.1 Docker を使う場合（推奨）

```bash
# プロジェクトルートで実行
docker compose up backend
```
*※ `db` コンテナが自動的に起動し、ヘルスチェック通過後にバックエンドが立ち上がります。*

### 7.2 直接起動する場合

ローカルマシンの PostgreSQL に直接接続する場合は、`.env` の接続先ホストを `db` から `localhost` に書き換える必要があります：
`DATABASE_URL=postgresql://postgres:postgres_password@localhost:5432/rag_chatbot`

```bash
cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 7.3 動作確認

```bash
curl http://localhost:8000/health
# 期待レスポンス: {"status":"ok","version":"0.1.0","services":{"database":"ok","llm":"ok"}}
```

ブラウザでも以下のローカル URL にアクセスして確認します。

| URL | 確認内容 |
|---|---|
| `http://localhost:8000/health` | バックエンド、DB、LLM 接続状態が JSON で表示されること |
| `http://localhost:8000/docs` | FastAPI の Swagger UI が表示されること |

Swagger UI: `http://localhost:8000/docs`


## 9. テストの実行

Windows では `pytest` コマンドが古い Anaconda など別環境の Python を参照する場合があるため、プロジェクトの仮想環境にある Python から `pytest` を実行します。

```powershell
cd backend

# 仮想環境が未作成の場合のみ
py -3.12 -m venv venv
.\venv\Scripts\python.exe -m pip install -r requirements.txt

# 全テスト実行
.\venv\Scripts\python.exe -m pytest tests/ -v

# カバレッジ付き
.\venv\Scripts\python.exe -m pytest tests/ -v --cov=app --cov-report=term-missing
```

macOS / Linux の場合は以下です。

```bash
cd backend

# 仮想環境が未作成の場合のみ
python3.12 -m venv venv
./venv/bin/python -m pip install -r requirements.txt

# 全テスト実行
./venv/bin/python -m pytest tests/ -v

# カバレッジ付き
./venv/bin/python -m pytest tests/ -v --cov=app --cov-report=term-missing
```

## 10. Ingest バッチの実行（テストデータ投入）

RAG の回答に使用するデータは、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) の手順に沿って投入します。
ここではローカル開発環境で最低限実行する流れを示します。データファイルの作り方、`sources.yaml` の項目、投入後のインデックス再構築や確認方法は RAG データ投入マニュアルを参照してください。

1. データファイルを `data/` ディレクトリに配置する。

```
data/
├── markdown/
│   └── sample.md
└── pdf/
    └── sample.pdf
```

2. `batch/sources.yaml` にメタ情報を記述する。

3. バッチを実行する。
```bash
# プロジェクトルートで実行
docker compose run --rm ingest
```

4. ベクトル検索インデックスは `db/01_schema.sql` で作成済みのため、通常は追加作業不要です。大量データ投入後にインデックスを再構築する場合は、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) の「5.1 ベクトル検索用インデックスの再構築」を参照してください。

5. データ投入後、ブラウザで `http://localhost:3000` を開き、投入したデータに関する質問を送信して回答と出典が表示されることを確認します。

## 11. 2回目以降・PC再起動後のローカル起動手順

初回セットアップが完了していれば、PC 再起動後や一度サービスを停止した後に、Docker イメージの再ビルド、テーブルスキーマの再作成、`npm install`、RAG データの再投入を毎回行う必要はありません。

### 11.1 前提

初回に以下が完了していることを確認します。

- Docker Desktop、Node.js をインストール済み
- プロジェクトルートに `.env` を作成済み
- `docker compose build` を実行済み
- `frontend` で `npm install` を実行済み
- 必要な RAG データを投入済み

### 11.2 Docker Desktop の起動

PC 再起動後は Docker Desktop を起動し、タスクトレイの表示が「Docker Desktop is running」になるまで待ちます。Docker Desktop が自動起動済みの場合、この操作は不要です。

次のコマンドで Docker Engine が利用できることを確認できます。

```powershell
docker info
```

### 11.3 DB とバックエンドの起動

PowerShell で `docker-compose.yml` があるプロジェクトルートへ移動してから実行します。プロジェクトの保存場所は環境ごとに異なるため、絶対パスは固定しません。

```powershell
# プロジェクトルートで実行
Test-Path .\docker-compose.yml  # True になることを確認

docker compose up -d backend
```

`backend` は `db` に依存しているため、このコマンドだけで DB も自動的に起動します。`docker compose up -d db` を先に実行する必要はありません。

起動状態を確認します。

```powershell
docker compose ps
```

`db` が `healthy`、`backend` が `Up` になれば起動完了です。同じコマンドを再度実行してもコンテナが重複して増えることはありません。

### 11.4 フロントエンドの起動

別の PowerShell を開き、プロジェクトルートから実行します。

```powershell
cd frontend
npm run dev
```

2回目以降は、`package.json` や `package-lock.json` が変更されていなければ `npm install` は不要です。

### 11.5 ブラウザでの確認

| URL | 確認内容 |
|---|---|
| `http://localhost:3000` | チャット画面が表示されること |
| `http://localhost:8000/health` | バックエンド、DB、LLM 接続状態が JSON で表示されること |
| `http://localhost:8000/docs` | FastAPI の Swagger UI が表示されること |

### 11.6 起動できない場合のログ確認

プロジェクトルートで実行します。

```powershell
docker compose logs --tail=100 backend
docker compose logs --tail=100 db
```

リアルタイムにログを追う場合は、コマンドへ `-f` を付けます。終了は `Ctrl+C` です。

### 11.7 作業終了時の停止

フロントエンドを起動した PowerShell で `Ctrl+C` を押します。その後、プロジェクトルートで Docker サービスを停止します。

```powershell
docker compose stop
```

コンテナ自体も削除する場合は、次を実行します。DB データを保存する Docker volume は残ります。

```powershell
docker compose down
```

> [!CAUTION]
> `docker compose down -v` は DB データを保存している volume も削除します。実行後はテーブル作成と RAG データ投入が再度必要になるため、DB を初期化したい場合以外は使用しないでください。

### 11.8 初回作業をやり直す必要があるケース

| 変更内容 | 必要な操作 |
|---|---|
| 通常の PC 再起動、サービスの再起動 | 第11章の起動手順のみ |
| `backend/requirements.txt` または Dockerfile を変更 | `docker compose up -d --build backend` |
| `frontend/package.json` または `package-lock.json` を変更 | `frontend` で `npm install` 後、`npm run dev` |
| `.env` を変更 | `docker compose up -d --force-recreate backend` |
| RAG 用 PDF / Markdown を追加・変更 | [14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md) に従って Ingest を再実行 |
| `docker compose down -v` を実行 | DB の初期化と RAG データ投入を再実行 |

DB データは Docker volume `pgdata` に保存されるため、通常の PC 再起動、`docker compose stop`、`docker compose down` では消えません。

## 12. トラブルシューティング
| 症状 | 原因 | 対処 |
|---|---|---|
| `/health` で `database: error` | DBコンテナが起動していない / パスワードの間違い | `docker compose ps` で稼働を確認。`.env` のパスワードが正しいか確認。 |
| `/health` で `llm: error` | `GEMINI_API_KEY` が正しくない / APIのクォータ制限 | キーを確認。または `USE_VERTEX_AI=true` の場合に認証情報が存在しない。 |
| バッチで `embedding` エラー | APIキー設定の不足 | `.env` に `GEMINI_API_KEY` が正しく設定されているか確認。 |
