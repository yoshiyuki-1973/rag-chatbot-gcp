# 12 テスト環境構築手順書（ローカル開発）

ローカル PC 上で開発・テストを行うための環境構築手順。
本番・検証環境の構築は [13_商用環境構築手順書.md](13_商用環境構築手順書.md) を参照。

---

## 1. 前提条件

### 1.1 必要ツール

| ツール | 用ンテージ | 確認コマンド | 用途 |
|---|---|---|---|
| Docker Desktop | 最新版 | `docker --version` | バックエンド・データベース・バッチの実行 |
| Node.js | 20.x 以上 | `node --version` | フロントエンド開発 |
| Python | 3.12 以上 | `python --version` | テスト直接実行時 |
| Git | 最新版 | `git --version` | ソースコード管理 |

### 1.2 必要アカウント・APIキー

ローカル開発環境では以下を用意します。

| サービス | 用途 | 注意 |
|---|---|---|
| Google AI Studio Gemini API | LLM 呼び出し / Embedding 生成 | 開発中も実 API を使用する（無料枠を使用） |

#### Google AI Studio（Gemini API）

1. [aistudio.google.com](https://aistudio.google.com) にサインインする
2. 左メニューの「API Keys」→「Create API Key」をクリック
3. Key name を入力（例: `rag-chatbot-dev`）し「Create」
4. 表示された API キーをコピーし、安全な場所に保存する

> [!NOTE]
> バックエンドのユニットテスト（pytest）はすべてモックを使用するため、テスト実行時に API キーは不要です。

## 3. テスト用ローカルデータベースのセットアップ

ローカル開発では Docker を使用して PostgreSQL (pgvector 拡張付き) を起動します。

### 3.1 データベースの起動

```bash
# データベースコンテナのみを起動
docker compose up -d db
```

### 3.2 テーブルスキーマの作成

データベースコンテナが起動したら、`psql` クライアントなどを利用してコンテナ内の PostgreSQL に接続し、テーブル定義を反映します。

```sql
-- pgvector 拡張の有効化
CREATE EXTENSION IF NOT EXISTS vector;

-- documents テーブル
CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    source_url      TEXT NOT NULL UNIQUE,
    file_type       TEXT NOT NULL CHECK (file_type IN ('markdown', 'pdf')),
    organization    TEXT,
    authority_score NUMERIC(3, 2) NOT NULL DEFAULT 0.50 CHECK (authority_score BETWEEN 0 AND 1),
    content_date    DATE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- document_chunks テーブル（768次元ベクトル）
CREATE TABLE document_chunks (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id   UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index   INTEGER NOT NULL,
    content       TEXT NOT NULL,
    embedding     VECTOR(768),
    token_count   INTEGER,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);

-- chat_history テーブル
CREATE TABLE chat_history (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id  TEXT NOT NULL,
    query       TEXT NOT NULL,
    response    TEXT NOT NULL,
    sources     JSONB NOT NULL DEFAULT '[]',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- updated_at 自動更新トリガー
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_documents_updated_at
BEFORE UPDATE ON documents
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- インデックス
CREATE INDEX idx_documents_organization ON documents (organization);
CREATE INDEX idx_chat_history_session_id ON chat_history (session_id);
CREATE INDEX idx_chat_history_created_at ON chat_history (created_at);
CREATE INDEX idx_document_chunks_embedding ON document_chunks USING ivfflat (embedding vector_cosine_ops) WITH (lists = 10);
```

## 4. 環境変数の設定

```bash
cp .env.example .env
```

`.env` を編集し、ローカル開発用の値を設定します。

```bash
# ── ローカルデータベース ──────────────────────────────────────────
# ホストOSから接続する場合は localhost を指定
# (コンテナ内から接続する場合は docker-compose によって自動的に `db` が名前解決されます)
DATABASE_URL=postgresql://postgres:postgres_password@db:5432/rag_chatbot

# ── LLM / Embedding (Google AI Studio) ─────────────────────────
GEMINI_API_KEY=your-gemini-api-key
GEMINI_MODEL=gemini-2.5-flash
GEMINI_FALLBACK_MODELS=gemini-2.5-flash-lite

# ── GCP / Vertex AI 設定 (ローカル開発では通常 false) ────────────────
USE_VERTEX_AI=false
GCP_PROJECT_ID=
GCP_LOCATION=us-central1

# ── アプリケーション設定 ─────────────────────────────────────────
LLM_PROVIDER=gemini
TOP_K_DEFAULT=5
CORS_ORIGINS=http://localhost:3000
DB_POOL_MIN_SIZE=1
DB_POOL_MAX_SIZE=3
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

## 5. バックエンド（FastAPI）の起動

### 5.1 Docker を使う場合（推奨）

```bash
docker compose up backend
```
*※ `db` コンテナが自動的に起動し、ヘルスチェック通過後にバックエンドが立ち上がります。*

### 5.2 直接起動する場合

ローカルマシンの PostgreSQL に直接接続する場合は、`.env` の接続先ホストを `db` から `localhost` に書き換える必要があります：
`DATABASE_URL=postgresql://postgres:postgres_password@localhost:5432/rag_chatbot`

```bash
cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 5.3 動作確認

```bash
curl http://localhost:8000/health
# 期待レスポンス: {"status":"ok","version":"0.1.0","services":{"database":"ok","llm":"ok"}}
```

Swagger UI: `http://localhost:8000/docs`

# カバレッジ付き
pytest tests/ -v --cov=app --cov-report=term-missing

## 8. Ingest バッチの実行（テストデータ投入）

1. データファイルを `data/` ディレクトリに配置する。

```
data/
├── markdown/
│   └── sample.md
└── pdf/
    └── sample.pdf
```

2. `batch/sources.yaml` にメタ情報を記述する。

3. バッチを実行する。
```bash
docker compose run --rm ingest
```

4. ベクトル検索インデックスを作成する（インジェスト完了後）。
```sql
CREATE INDEX idx_document_chunks_embedding
ON document_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 10);
```

## 9. トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| `/health` で `database: error` | DBコンテナが起動していない / パスワードの間違い | `docker compose ps` で稼働を確認。`.env` のパスワードが正しいか確認。 |
| `/health` で `llm: error` | `GEMINI_API_KEY` が正しくない / APIのクォータ制限 | キーを確認。または `USE_VERTEX_AI=true` の場合に認証情報が存在しない。 |
| バッチで `embedding` エラー | APIキー設定の不足 | `.env` に `GEMINI_API_KEY` が正しく設定されているか確認。 |
