# 04 詳細設計書

本ドキュメントは、RAG チャットボットの「どのように実現するか（内部設計）」を定義する。
外部仕様は [02_基本設計書.md](02_基本設計書.md) を参照すること。

## 1. レイヤー構成と責務

### 1.1 バックエンド（FastAPI）のレイヤー構成

```
┌─────────────────────────────────────────────────────┐
│ Router 層（routers/）                                │
│  - HTTP リクエストの受付・バリデーション（Pydantic） │
│  - レスポンスモデルの返却                            │
│  - ビジネスロジックは持たない                        │
├─────────────────────────────────────────────────────┤
│ Service 層（services/）                              │
│  - RAG パイプラインの制御（RAGService）              │
│  - Embedding 生成（EmbeddingClient）                 │
│  - エラーを HTTPException に変換する唯一の層         │
├─────────────────────────────────────────────────────┤
│ Adapter 層（adapters/）                              │
│  - 外部依存を抽象化（LLM・VectorStore）             │
│  - インターフェース（BaseLLMClient・BaseVectorStore）│
│  - 具体実装は交換可能                                │
├─────────────────────────────────────────────────────┤
│ インフラ層（main.py / settings.py）                  │
│  - 起動時の依存オブジェクト生成（ファクトリ）        │
│  - 環境変数管理（pydantic-settings）                 │
│  - DB コネクションプール管理（asyncpg）              │
└─────────────────────────────────────────────────────┘
```

### 1.2 バッチ（Ingest Pipeline）のレイヤー構成

```
ingest.py（エントリポイント）
  │
  ├── extractors/  テキスト抽出（Markdown・PDF）
  ├── chunker.py   チャンク分割（token ベース）
  ├── embedder.py  Embedding 生成（Vertex AI / Gemini API バッチ）
  └── db.py        DB 登録（asyncpg・トランザクション）
```

## 2. コンポーネント設計

### 2.1 RAGService

RAG パイプライン全体を制御するサービス。

```
RAGService
├── embedder: EmbeddingClient      クエリをベクトル化
├── vector_store: BaseVectorStore  類似チャンクを検索
└── llm: BaseLLMClient             コンテキスト付きで回答生成

主要メソッド:
  chat(query, session_id, top_k) → ChatResponse
    1. embedder.embed(query) でクエリをベクトル化
    2. vector_store.search(embedding, top_k) でチャンクを取得
    3. _build_context(chunks) でプロンプトのコンテキストを組み立てる
    4. llm.generate(system_prompt, query) で回答を生成
    5. vector_store.save_chat_history(...) で履歴を保存（失敗しても無視）
    6. ChatResponse を返す

  search(query, top_k, min_similarity, min_authority_score) → SearchResponse
    1. embedder.embed(query) でクエリをベクトル化
    2. vector_store.search(...) でチャンクを取得
    3. SearchResponse を返す

  _build_context(chunks) → str
    - チャンクが空の場合: "" を返す（LLM に「情報なし」を伝えさせる）
    - チャンクがある場合: "[出典: タイトル]\n本文" を "---" で結合
```

### 2.2 LLM Adapter

LLM の差し替えを可能にする抽象層。

```
BaseLLMClient（ABC）
  abstract generate(system_prompt, user_message, max_tokens, temperature) → str
  default  stream(...)  → AsyncIterator[str]  # generate を1回 yield するデフォルト実装

GeminiLLMClient（BaseLLMClient の具体実装）
  - google-genai SDK を使用して Vertex AI / Gemini API に接続
  - model: gemini-2.5-flash（settings.GEMINI_MODEL で変更可）
  - generate(): None の場合は "" を返す（or "" でガード）
```

新規 LLM 追加時は `BaseLLMClient` を継承し `generate` を実装するだけでよい。

### 2.3 Vector Store Adapter

ベクトル検索の差し替えを可能にする抽象層。

```
BaseVectorStore（ABC）
  abstract search(query_embedding, top_k, min_similarity, min_authority_score) → list[SearchResult]
  abstract save_chat_history(session_id, query, response, sources) → None

PostgreSQLVectorStore（BaseVectorStore の具体実装）
  - asyncpg.Pool 経由で PostgreSQL / Cloud SQL に接続
  - search(): サブクエリで IVFFlat インデックスを活用しコサイン距離を1回だけ評価
  - save_chat_history(): session_id が None の場合は保存をスキップ
```

## 3. RAG パイプライン詳細設計

### 3.1 チャット処理シーケンス

```
User          Next.js        FastAPI(RAG)      Embedding    pgvector    Gemini API
 │                │                │               │             │           │
 │  質問入力      │                │               │             │           │
 │──────────────►│                │               │             │           │
 │               │  POST /chat    │               │             │           │
 │               │───────────────►│               │             │           │
 │               │                │  embed(query) │             │           │
 │               │                │──────────────►│             │           │
 │               │                │◄──────────────│             │           │
 │               │                │  vector(768)  │             │           │
 │               │                │               │             │           │
 │               │                │       search(embedding, k) │           │
 │               │                │──────────────────────────►│           │
 │               │                │◄──────────────────────────│           │
 │               │                │     chunks + metadata      │           │
 │               │                │                            │           │
 │               │                │      generate(ctx, query)  │           │
 │               │                │──────────────────────────────────────►│
 │               │                │◄──────────────────────────────────────│
 │               │                │      answer text           │           │
 │               │                │                            │           │
 │               │  {answer, sources}             │             │           │
 │               │◄───────────────│               │             │           │
 │  回答・出典表示│                │               │             │           │
 │◄──────────────│                │               │             │           │
```

### 3.2 Ingest パイプラインシーケンス

```
ingest.py
  │
  ├─ sources.yaml を読み込む
  │
  └─ for each source:
       │
       ├─ extract_text()
       │    Markdown: path.read_text(encoding="utf-8")
       │    PDF: PyMuPDF で全ページテキストを [page:N] メタ付きで結合
       │
       ├─ split_text()
       │    段落単位（\n\n で分割）で chunk_size=500 token に収まるよう結合
       │    長い段落は文末（。.!?）で再分割 + tail overlap=50 token
       │
       ├─ embed_many() バッチサイズ 100 件ずつ / リトライ最大 3 回
       │
       └─ upsert_document_with_chunks() トランザクション内で:
            1. documents を source_url で upsert
            2. document_chunks を DELETE してから executemany で一括 insert
```

### 3.3 プロンプト設計

#### システムプロンプト

```
あなたはスポーツに関する質問に答えるアシスタントです。
以下のルールに従ってください。

1. 必ず提供されたコンテキストの情報のみを使用して回答してください。
2. コンテキストに情報がない場合は「提供された情報の中には
   該当する情報が見つかりませんでした」と回答してください。
3. 回答は日本語で行ってください。
4. 推測や創作は行わないでください。
5. 回答の根拠となったドキュメントは別途出典として提示されます。

コンテキスト:
{context}
```

#### コンテキスト組み立て規則

- チャンクが 0 件: `context = ""`（空文字列）→ LLM が「見つかりませんでした」と回答する
- チャンクが N 件: `"[出典: {title}]\n{content}"` を `"\n\n---\n\n"` で結合
- top_k はデフォルト 5 件（`settings.TOP_K_DEFAULT` で変更可）
- temperature: 0.1（正確性優先）

## 4. ベクトル検索詳細設計

### 4.1 SQL 設計方針

```sql
-- サブクエリで IVFFlat インデックスを活用（ORDER BY + LIMIT パターン）
-- コサイン距離は内側で 1 回だけ評価し、外側で min_similarity フィルターをかける
SELECT * FROM (
    SELECT
        c.document_id::text,
        c.chunk_index,
        c.content,
        1 - (c.embedding <=> $1::vector) AS similarity,
        d.title, d.source_url, d.organization,
        d.authority_score::float
    FROM document_chunks c
    JOIN documents d ON c.document_id = d.id
    WHERE c.embedding IS NOT NULL
      AND d.authority_score >= $3
    ORDER BY c.embedding <=> $1::vector   -- IVFFlat インデックス使用
    LIMIT $2                               -- top_k
) ranked
WHERE ranked.similarity >= $4             -- min_similarity 後フィルター
ORDER BY ranked.similarity DESC
```

### 4.2 IVFFlat インデックス設定

| パラメータ | 値 | 理由 |
|---|---|---|
| lists | 10（初期）/ 100（1万件超） | `sqrt(行数)` が目安。初期データ量では 10 が適切 |
| 距離関数 | `vector_cosine_ops` | テキスト embedding はコサイン類似度が最適 |
| インデックス作成タイミング | 全チャンク insert 後 | 空テーブルへの作成は無意味 |

## 5. エラーハンドリング設計

### 5.1 エラー分類と HTTP ステータス

| エラー種別 | 発生箇所 | HTTP ステータス | エラーコード |
|---|---|---|---|
| リクエストバリデーション失敗 | Pydantic | 422 | VALIDATION_ERROR |
| Embedding API 失敗・DB 接続失敗 | RAGService.search | 500 | SEARCH_ERROR |
| LLM API 失敗・タイムアウト | RAGService.chat | 500 | LLM_ERROR |
| 全サービス障害 | /health エンドポイント | 503 | — |

### 5.2 エラーレスポンス形式

```json
{
  "detail": {
    "code": "LLM_ERROR",
    "message": "LLM サービスへの接続に失敗しました。しばらく後にお試しください。"
  }
}
```

### 5.3 サイレントエラーの扱い

以下はエラーが発生しても主処理を止めない（ログのみ出力）。

| 処理 | 理由 |
|---|---|
| 会話履歴の保存（`save_chat_history`） | 履歴保存失敗が回答配信の妨げになるべきでない |
| バッチ内の個別ファイル処理失敗 | 1 件の失敗で全件停止を避ける |

## 6. ログ設計

### 6.1 ログレベル方針

| レベル | 用途 |
|---|---|
| INFO | 正常なリクエスト処理（エンドポイント・レイテンシ） |
| WARNING | サイレントエラー（履歴保存失敗等） |
| ERROR | ユーザーに影響するエラー（LLM_ERROR・SEARCH_ERROR） |

### 6.2 ログフォーマット（/chat）

```json
{
  "timestamp": "2026-04-25T10:00:00Z",
  "level": "INFO",
  "endpoint": "/chat",
  "query_length": 20,
  "retrieved_chunks": 5,
  "llm_latency_ms": 1230,
  "total_latency_ms": 1450
}
```

スタックトレースはログにのみ出力し、レスポンスには含めない。

## 7. セキュリティ設計

### 7.1 API キー・クレデンシャル管理

- 外部サービス API キーや GCP サービスアカウント認証情報は環境変数で管理する
- コードへのハードコードは禁止
- `.env` は `.gitignore` に追加し Git 管理外とする
- `.env.example` でキー名と説明のみを管理する

### 7.2 CORS 設定

```python
allow_origins = settings.cors_origin_list  # カンマ区切りで複数オリジン設定可
allow_credentials = True
allow_methods = ["*"]
allow_headers = ["*"]
```

本番では CORS_ORIGINS でデプロイ済みのフロントエンド Cloud Run の URL のみ許可する。

### 7.3 入力バリデーション（Pydantic）

| フィールド | 制約 |
|---|---|
| query | min_length=1, max_length=500 |
| top_k | ge=1, le=20（None の場合は settings.TOP_K_DEFAULT を使用） |
| min_similarity | ge=-1, le=1 |
| min_authority_score | ge=0, le=1 |
| authority_score | ge=0, le=1 |
| TOP_K_DEFAULT（設定値） | 1〜20（@field_validator で検証） |

## 8. 起動・初期化設計

### 8.1 lifespan による起動シーケンス

```python
@asynccontextmanager
async def lifespan(app):
    # 1. 必須環境変数チェック（DATABASE_URL、USE_VERTEX_AI設定による各APIキー/プロジェクトID）
    # 2. asyncpg.create_pool() でコネクションプールを確立
    # ▼ try ブロック（create_pool 成功後に確実にクローズするため）
    # 3. EmbeddingClient 生成（google-genai SDK の初期化）
    # 4. _create_llm_client(settings) で LLM クライアントを生成
    # 5. app.state にオブジェクトをセット
    # 6. yield（リクエスト処理）
    # ▼ finally
    # 7. db_pool.close()
```

`create_pool` より後の初期化が失敗しても `finally` でプールがクローズされる。

### 8.2 LLM ファクトリ（_create_llm_client）

```python
def _create_llm_client(settings) -> BaseLLMClient:
    if settings.llm_provider == "gemini":
        return GeminiLLMClient(
            api_key=settings.gemini_api_key,
            models=settings.gemini_model_candidates,
            vertexai=settings.use_vertex_ai,
            project=settings.gcp_project_id,
            location=settings.gcp_location
        )
    raise RuntimeError(f"Unsupported LLM_PROVIDER: {settings.llm_provider!r}")
```

新しいプロバイダー追加時は、ここに `elif` を追加し対応 Adapter を実装する。

### 8.3 依存性注入（DI）

```
app.state
  ├── db_pool     asyncpg.Pool（接続プール）
  ├── embedder    EmbeddingClient（全リクエスト共有）
  ├── llm         BaseLLMClient の具体実装（全リクエスト共有）
  ├── llm_health_status   str | None（ヘルスチェックキャッシュ）
  └── llm_health_checked_at  float（キャッシュ更新タイムスタンプ）

get_rag_service(request) → RAGService
  リクエストごとに RAGService を生成し、
  PostgreSQLVectorStore(request.app.state.db_pool) を渡す
```

## 9. ヘルスチェック設計

### 9.1 チェック内容

| 対象 | 方法 | 障害時の status |
|---|---|---|
| DB（Cloud SQL） | `SELECT 1` を実行 | `"error"` |
| LLM（Gemini） | `generate("Return ok.", "ping", max_tokens=1)` を 2 秒以内に完了 | `"error"` |

### 9.2 LLM チェックのキャッシュ

- TTL: 30 秒
- 成功時: `llm_health_status = "ok"` をキャッシュ
- 失敗時: `llm_health_status = "error"` をキャッシュしない（毎リクエスト再試行）
  - 理由: LLM 障害は一時的なことが多く、次のリクエストで回復する可能性がある

### 9.3 全体ステータス判定

```
any "error"  → overall = "error"  → HTTP 503
all "ok"     → overall = "ok"     → HTTP 200
otherwise    → overall = "degraded" → HTTP 200
```
