## 2. documents テーブル

### 論理名: ドキュメント

取り込んだドキュメントのメタ情報を管理するテーブル。本文テキストは保持せず、チャンク側に持たせる。

```sql
CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    source_url      TEXT NOT NULL UNIQUE,
    file_type       TEXT NOT NULL CHECK (file_type IN ('markdown', 'pdf')),
    organization    TEXT,
    authority_score NUMERIC(3, 2) NOT NULL DEFAULT 0.50 CHECK (authority_score BETWEEN 0 AND 1),
    content_date    DATE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### カラム定義

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---|---|---|---|---|
| id | UUID | NOT NULL | gen_random_uuid() | 主キー。UUIDv4 を自動生成 |
| title | TEXT | NOT NULL | — | ドキュメントのタイトル。出典表示に使用する |
| source_url | TEXT | NOT NULL | — | ドキュメントの元 URL またはファイルパス。UNIQUE 制約で重複登録を防ぐ |
| file_type | TEXT | NOT NULL | — | `markdown` または `pdf` のみ許容 |
| organization | TEXT | NULL | — | 発行元組織名（例: JOC、FIFA など）。NULL の場合は「不明」として扱う |
| authority_score | NUMERIC(3,2) | NOT NULL | 0.50 | 0〜1 の信頼性スコア。公的機関は高く設定する（詳細は §5 メタ情報設計を参照） |
| content_date | DATE | NULL | — | ドキュメントの作成・更新日。NULL の場合は日付不明として扱う |
| created_at | TIMESTAMPTZ | NOT NULL | now() | レコード作成日時 |
| updated_at | TIMESTAMPTZ | NOT NULL | now() | レコード更新日時。下記トリガーで UPDATE 時に自動更新される |

### updated_at 自動更新トリガー

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_documents_updated_at
BEFORE UPDATE ON documents
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

### インデックス

| インデックス名 | カラム | 種類 | 用途 |
|---|---|---|---|
| documents_pkey | id | PRIMARY KEY | 主キー検索 |
| documents_source_url_key | source_url | UNIQUE | 重複登録防止・upsert のキー |
| idx_documents_organization | organization | BTREE | 組織別検索 |

## 3. document_chunks テーブル

### 論理名: ドキュメントチャンク

ドキュメントを分割したチャンク単位のテキストと embedding ベクトルを管理するテーブル。ベクトル検索の実体はこのテーブルに対して行う。

```sql
-- pgvector 拡張が必要
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE document_chunks (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id   UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index   INTEGER NOT NULL,
    content       TEXT NOT NULL,
    embedding     VECTOR(768),
    token_count   INTEGER,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);
```

### カラム定義

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---|---|---|---|---|
| id | UUID | NOT NULL | gen_random_uuid() | 主キー |
| document_id | UUID | NOT NULL | — | documents.id への外部キー。親ドキュメントが削除されれば CASCADE 削除される |
| chunk_index | INTEGER | NOT NULL | — | ドキュメント内のチャンク順序（0 始まり）。同一ドキュメント内で UNIQUE |
| content | TEXT | NOT NULL | — | チャンクのテキスト本文 |
| embedding | VECTOR(768) | NULL | — | Vertex AI / Gemini API `gemini-embedding-001`（768 次元）のベクトル。NULL は未生成を示す |
| token_count | INTEGER | NULL | — | チャンクのトークン数。デバッグ・コスト計算に使用 |
| created_at | TIMESTAMPTZ | NOT NULL | now() | レコード作成日時 |

### インデックス

| インデックス名 | カラム | 種類 | 用途 |
|---|---|---|---|
| document_chunks_pkey | id | PRIMARY KEY | 主キー検索 |
| document_chunks_document_id_chunk_index_key | (document_id, chunk_index) | UNIQUE | チャンク順序管理・重複防止 |
| idx_document_chunks_embedding | embedding | IVFFlat (lists=10) | ベクトル近傍検索（コサイン距離） |

```sql
-- IVFFlat インデックス（embedding 列全件 insert 後に作成すること）
-- lists は sqrt(行数) 程度が推奨。初期は 10 で十分、1万件超で 100 に上げる
CREATE INDEX idx_document_chunks_embedding
ON document_chunks USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 10);
```

> **補足**: IVFFlat は近似最近傍検索のインデックス。行数が 0 件の状態でインデックスを作成すると空のインデックスになるため、チャンクを全件 insert した後に実行すること。

## 4. chat_history テーブル

### 論理名: 会話履歴

ユーザーの質問・回答・参照出典を記録するテーブル。MVP では参照のみで LLM への再入力はしないが、将来の文脈維持機能のために保持する。

```sql
CREATE TABLE chat_history (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id  TEXT NOT NULL,
    query       TEXT NOT NULL,
    response    TEXT NOT NULL,
    sources     JSONB NOT NULL DEFAULT '[]',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### カラム定義

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---|---|---|---|---|
| id | UUID | NOT NULL | gen_random_uuid() | 主キー |
| session_id | TEXT | NOT NULL | — | セッション識別子（フロントエンドで生成した UUID）。複数ターンの会話を束ねる |
| query | TEXT | NOT NULL | — | ユーザーが入力したクエリ原文 |
| response | TEXT | NOT NULL | — | LLM が生成した回答テキスト |
| sources | JSONB | NOT NULL | [] | 参照した出典情報の配列（詳細は §10 出典表示設計を参照） |
| created_at | TIMESTAMPTZ | NOT NULL | now() | 会話が発生した日時 |

### sources JSONB の構造

```json
[
  {
    "document_id": "uuid",
    "title": "FIFA ワールドカップ 2022 技術レポート",
    "source_url": "https://example.com/report.pdf",
    "organization": "FIFA",
    "authority_score": 0.9,
    "chunk_index": 3,
    "similarity": 0.87
  }
]
```

### インデックス

| インデックス名 | カラム | 種類 | 用途 |
|---|---|---|---|
| chat_history_pkey | id | PRIMARY KEY | 主キー検索 |
| idx_chat_history_session_id | session_id | BTREE | セッション別履歴取得 |
| idx_chat_history_created_at | created_at | BTREE | 時系列ソート |

## 5. テーブル間のリレーション

| リレーション | 種類 | 説明 |
|---|---|---|
| documents → document_chunks | 1:N | 1 つのドキュメントは複数のチャンクを持つ |
| documents → chat_history.sources | 1:N（JSONB 参照） | 1 つのドキュメントは複数の会話履歴で参照される |

## 6. メタ情報設計

### 6.1 必須メタ情報項目

| 項目 | カラム | 必須/任意 | 説明 |
|---|---|---|---|
| ドキュメントタイトル | title | 必須 | 出典表示に直接使われるため必須 |
| 元 URL / ファイルパス | source_url | 必須 | 重複排除キー。公開 URL があればリンク表示に使う |
| ファイル種別 | file_type | 必須 | パーサーの選択に使用 |
| 発行元組織 | organization | 任意 | 権威スコアの計算根拠として使用 |
| 信頼性スコア | authority_score | 必須 | 0〜1 の数値。出典の信頼度を表す |
| コンテンツ日付 | content_date | 任意 | 情報の新しさを示すために使用 |

### 6.2 authority_score の設計

`authority_score` は 0.0〜1.0 の数値で、情報ソースの権威・信頼性を表す。

#### スコアの目安

| スコア範囲 | 対象の例 | 理由 |
|---|---|---|
| 0.90 〜 1.00 | JOC・FIFA・IOC 等の公式国際機関 | 一次情報源であり、正確性が高い |
| 0.70 〜 0.89 | 各競技の公式連盟・協会 | 競技固有の公式情報だが国際機関よりは範囲が狭い |
| 0.50 〜 0.69 | スポーツ専門メディア・大手報道機関 | 専門性はあるが編集バイアスが入りうる |
| 0.30 〜 0.49 | 一般ブログ・Wikipedia | 情報の正確性にばらつきがある |
| 0.00 〜 0.29 | 出所不明・個人サイト | 信頼性が低い |

#### authority_score の用途

1. **出典表示**: UI でスコアをバッジ（「高信頼」「中信頼」等）として表示する
2. **検索結果のランキング調整（将来）**: 類似度スコアに authority_score を重み付けして最終スコアを算出できる（`final_score = similarity * 0.8 + authority_score * 0.2`）
3. **フィルタリング**: 低信頼ソースを除外するオプションを設けられる

#### スコアの設定方法

- バッチ取り込み時に `ingest.py` の設定ファイル（`sources.yaml`）で各ソースの `authority_score` を手動設定する
- 自動化は MVP のスコープ外とし、将来の拡張として検討する
