# 15 運用マニュアル（GCP運用ガイド）

GCP上に移行したRAGチャットボットシステム（ステージング/本番）のデプロイ構成、環境変数、監視、およびメンテナンス手順を記載。

---

## 1. デプロイ構成

### 1.1 全体構成

| サービス | ホスティング先 | プラン | 用途 |
|---|---|---|---|
| Next.js フロントエンド | Cloud Run | 従量課金 | UI 配信・SSR |
| FastAPI バックエンド | Cloud Run | 従量課金 | API サーバー |
| PostgreSQL + pgvector | Cloud SQL for PostgreSQL | 従量課金 | DB・ベクトル検索 |
| Ingest バッチ | ローカル Docker | — | データ取り込み (Cloud SQL Auth Proxy 経由) |

---

### 1.2 Cloud Run (フロントエンド・バックエンド)

#### 自動デプロイと CI/CD

通常、GitHub リポジトリと Google Cloud Build を連携させて CI/CD を構築します。
- `main` ブランチへのマージ: 本番環境 (`rag-chatbot-gcp-prod`) に自動デプロイ
- `develop` ブランチへのマージ: 検証環境 (`rag-chatbot-gcp-stg`) に自動デプロイ

#### スケーリング設定

Cloud Run サービスは、コールドスタートを防ぎ、コストを最適化するために以下のスケーリングパラメータを設定可能です。

- **最小インスタンス数 (`--min-instances`)**: 
  - `stg` / ポートフォリオ用: `0`（リクエストがない時は無料）
  - `prod` / 常時応答用: `1`（コールドスタート防止）
- **最大インスタンス数 (`--max-instances`)**:
  - 予算超過を防ぐために `3` 〜 `5` 程度に制限することを推奨。

---

### 1.3 Cloud SQL (PostgreSQL)

#### メンテナンスとバックアップ

Cloud SQL は以下のマネージド機能を提供します。

- **自動バックアップ**: 毎日自動的に取得されます。保存期間はデフォルトで 7 日間です。
- **メンテナンスウィンドウ**: システムアップデートに伴う一時停止（通常数秒）が発生する時間帯を、深夜帯などにスケジュール指定することを推奨します。

---

### 1.4 データ追加・更新手順 (ローカル Ingest)

#### データの追加・更新フロー

1. データファイルを `data/markdown/` または `data/pdf/` に追加・配置する。
2. `batch/sources.yaml` に対象ファイルのメタ情報（タイトル、元URL、権威スコア等）を追記する。
3. `gcloud config set project [PROJECT_ID]` で、更新対象のプロジェクト（`stg` または `prod`）を設定する。
4. **Cloud SQL Auth Proxy** をIngestと同じDockerネットワークで起動する。
   ```powershell
   # Stagingの場合
   $env:APP_ENV_FILE = ".env.stg"

   # Productionの場合
   # $env:APP_ENV_FILE = ".env.prod"

   $TERRAFORM_EXE = "C:\apps\terraform_1.15.7\terraform.exe"
   $CONNECTION_NAME_OUTPUT = & $TERRAFORM_EXE "-chdir=terraform" output -raw db_connection_name
   if ($LASTEXITCODE -ne 0) {
     throw "TerraformからCloud SQL接続名を取得できません。"
   }
   $env:CLOUD_SQL_CONNECTION_NAME = $CONNECTION_NAME_OUTPUT.Trim()

   docker compose -f docker-compose.yml -f docker-compose.gcp.yml up -d --force-recreate cloud-sql-proxy
   docker compose -f docker-compose.yml -f docker-compose.gcp.yml logs --tail 20 cloud-sql-proxy
   ```
5. 対象環境の環境変数ファイルで `DATABASE_URL` が `cloud-sql-proxy:5432` を指していることを確認し、バッチを実行する。
   ```powershell
   docker compose -f docker-compose.yml -f docker-compose.gcp.yml build ingest

   docker compose -f docker-compose.yml -f docker-compose.gcp.yml run --rm --no-deps `
     -v "${env:APPDATA}/gcloud/application_default_credentials.json:/tmp/gcp-adc.json:ro" `
     -e GOOGLE_APPLICATION_CREDENTIALS=/tmp/gcp-adc.json `
     ingest

   docker compose -f docker-compose.yml -f docker-compose.gcp.yml stop cloud-sql-proxy
   Remove-Item Env:APP_ENV_FILE -ErrorAction SilentlyContinue
   Remove-Item Env:CLOUD_SQL_CONNECTION_NAME -ErrorAction SilentlyContinue
   ```
6. 最初にDB事前確認の成功、最後に `0 files failed` が表示されたことを確認する。
7. **インデックスの再作成**: データを大量に追加した場合、検索速度向上のためにインデックス（IVFFlat / HNSW）を再ビルドします。

## 2. 環境変数一覧

### 2.1 バックエンド (FastAPI)

| 変数名 | 必須 | 説明 | 設定例 |
|---|---|---|---|
| DATABASE_URL | 必須 | PostgreSQL 接続文字列 | `postgresql://app_user:pwd@/rag_chatbot?host=/cloudsql/project:region:instance` (本番) / `postgresql://postgres:pwd@db:5432/rag_chatbot` (ローカル) |
| GEMINI_API_KEY | 任意 | Google AI Studio 用の API キー (Vertex AI を使用しない場合のみ必須) | `AIzaSy...` |
| GEMINI_MODEL | 任意 | デフォルトで使用する LLM モデル | `gemini-2.5-flash` |
| GEMINI_FALLBACK_MODELS | 任意 | LLM 高負荷時のフォールバックモデル | `gemini-2.5-flash-lite` |
| USE_VERTEX_AI | 必須 | Vertex AI を優先して使用するかどうか | `true` (本番・検証) / `false` (ローカル開発用) |
| GCP_PROJECT_ID | 任意 | GCP プロジェクト ID (Vertex AI 使用時必須) | `rag-chatbot-gcp-prod` |
| GCP_LOCATION | 任意 | GCP リージョン (Vertex AI 使用時必須) | `asia-northeast1` |
| LLM_PROVIDER | 任意 | 使用する LLM プロバイダー | `gemini` |
| TOP_K_DEFAULT | 任意 | 検索時のデフォルト取得チャンク数 | `10` |
| CORS_ORIGINS | 任意 | CORS 許可オリジン | `https://frontend-url.run.app,http://localhost:3000` |

### 2.2 フロントエンド (Next.js)

| 変数名 | 必須 | 説明 | 例 |
|---|---|---|---|
| NEXT_PUBLIC_API_BASE_URL | 必須 | バックエンド API のエンドポイント URL | `https://rag-chatbot-gcp-backend-xxx-an.a.run.app` |

## 3. 監視・ログ・アラート

Google Cloud が提供する **Cloud Operations (旧 Stackdriver)** を利用します。

### 3.1 ログ確認 (Cloud Logging)

アプリケーションのログやエラー追跡は、Cloud Console の **Cloud Logging** で一元管理されます。

- **バックエンドログ**: Cloud Run の `rag-chatbot-gcp-backend` ログを確認
- **データベースログ**: Cloud SQL の `rag-db` ログを確認
- **クエリフィルタ例 (エラーログの抽出)**:
  ```
  resource.type="cloud_run_revision"
  severity>=ERROR
  ```

### 3.2 可用性監視 (Cloud Monitoring / Uptime Checks)

システムの稼働状況を監視するために、Cloud Monitoring で **Uptime Checks (稼働時間チェック)** を設定することを推奨します。

- **ターゲット**: バックエンドの `https://<backend-url>/health` エンドポイント
- **頻度**: 1 分〜5 分ごと
- **アラート**: 連続して 3 回以上チェックが失敗した場合、メールや Slack にアラート通知するポリシーを作成します。

> [!IMPORTANT]
> `/health` のDB判定は接続確認（`SELECT 1`）です。必須テーブルやRAGデータ件数の監視を兼ねません。デプロイ後は、[14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md)に従い、3テーブルの存在と `documents`・`document_chunks` の件数も確認してください。

## 4. トラブルシューティング

| 症状 | 考えられる原因 | 対策 |
|---|---|---|
| API `/health` で `database: error` | 1. Cloud SQL インスタンスが停止している<br>2. サービスアカウントの権限不足<br>3. 接続文字列の間違い | 1. GCP コンソールで Cloud SQL の稼働状況を確認<br>2. Cloud Run に設定されたサービスアカウントが `Cloud SQL クライアント` ロールを持っているか確認<br>3. `DATABASE_URL` 設定を確認 |
| API `/health` で `llm: error` | 1. `GCP_PROJECT_ID` が間違っている<br>2. Vertex AI API が有効化されていない<br>3. サービスアカウントに `Vertex AI ユーザー` ロールがない | 1. 環境変数の設定を確認<br>2. GCP プロジェクトで API を有効化<br>3. サービスアカウントの IAM ロール設定を確認 |
| 画面に `Failed to fetch` と表示される | フロントエンドの接続先URLまたはCORS不一致 | フロントエンドのビルド時バックエンドURL、Terraformの `cors_origins`、Cloud Runログの `Disallowed CORS origin` を確認する |
| 画面に「ベクトル検索に失敗しました」と表示される | スキーマ未作成、データ未投入、Embedding・DB検索エラー | まずCloud Runバックエンドログを確認する。`UndefinedTableError`ならスキーマを作成し、テーブルがある場合は文書・チャンク件数を確認する |
| 検索でエラーになり、テーブルとデータは存在する | インジェスト時のベクトル次元が合っていない | DB の `document_chunks.embedding` が `VECTOR(768)` になっているか確認し、合っていない場合はスキーマを再作成してデータを再インジェストする |
| Ingestで `relation "document_chunks" does not exist` | 対象環境のスキーマ未作成、接続先DBの誤り | [14_RAGデータ投入マニュアル.md](14_RAGデータ投入マニュアル.md)の初回スキーマ作成を実行する |
| Docker APIへ接続できない | Docker Desktopが停止中 | Docker Desktopを起動し、`docker info` の成功後に再実行する |
| GCP投入時にローカルDBも起動する | GCP用Composeファイルまたは`--no-deps`の指定漏れ | `docker compose -f docker-compose.yml -f docker-compose.gcp.yml run --rm --no-deps ingest`を使用する |
| Ingestが`[Errno -2] Name or service not known`で全件失敗する | `host.docker.internal`を名前解決できない | 環境別ファイルを`cloud-sql-proxy:5432`へ変更し、Docker内Proxyを起動して再実行する |
