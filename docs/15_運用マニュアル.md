# 14 運用マニュアル (GCP運用ガイド)

GCP上に移行したRAGチャットボットシステム（ステージング/本番）のデプロイ構成、環境変数、監視、およびメンテナンス手順を記載。

---

## 1. デプロイ構成

### 1.1 全体構成

| サービス | ホスティング先 | プラン | 用途 |
|---|---|---|---|
| Next.js フロントエンド | Cloud Run | 従量課金 | UI 配信・SSR |
| FastAPI バックエンド | Cloud Run | 従量課金 | API サーバー |
| PostgreSQL + pgvector | Cloud SQL for PostgreSQL | 従量課金 | DB・ベクトル検索 |
| Ingest バッチ | ローカル Docker | — | データ取り込み (Cloud SQL Auth Proxy 経由) |

---

### 1.2 Cloud Run (フロントエンド・バックエンド)

#### 自動デプロイと CI/CD

通常、GitHub リポジトリと Google Cloud Build を連携させて CI/CD を構築します。
- `main` ブランチへのマージ: 本番環境 (`rag-chatbot-prod`) に自動デプロイ
- `develop` ブランチへのマージ: 検証環境 (`rag-chatbot-stg`) に自動デプロイ

#### スケーリング設定

Cloud Run サービスは、コールドスタートを防ぎ、コストを最適化するために以下のスケーリングパラメータを設定可能です。

- **最小インスタンス数 (`--min-instances`)**: 
  - `stg` / ポートフォリオ用: `0`（リクエストがない時は無料）
  - `prod` / 常時応答用: `1`（コールドスタート防止）
- **最大インスタンス数 (`--max-instances`)**:
  - 予算超過を防ぐために `3` 〜 `5` 程度に制限することを推奨。

---

### 1.3 Cloud SQL (PostgreSQL)

#### メンテナンスとバックアップ

Cloud SQL は以下のマネージド機能を提供します。

- **自動バックアップ**: 毎日自動的に取得されます。保存期間はデフォルトで 7 日間です。
- **メンテナンスウィンドウ**: システムアップデートに伴う一時停止（通常数秒）が発生する時間帯を、深夜帯などにスケジュール指定することを推奨します。

---

### 1.4 データ追加・更新手順 (ローカル Ingest)

#### データの追加・更新フロー

1. データファイルを `data/markdown/` または `data/pdf/` に追加・配置する。
2. `batch/sources.yaml` に対象ファイルのメタ情報（タイトル、元URL、権威スコア等）を追記する。
3. `gcloud config set project [PROJECT_ID]` で、更新対象のプロジェクト（`stg` または `prod`）を設定する。
4. **Cloud SQL Auth Proxy** を起動して、DBへ接続可能な状態にする。
   ```bash
   cloud-sql-proxy --port 5432 $(gcloud config get-value project):asia-northeast1:rag-db
   ```
5. ローカルからバッチを実行し、`text-embedding-004` (768次元) で Embedding を生成して DB に格納する。
   ```bash
   docker compose --env-file .env.prod run --rm ingest
   ```
6. **インデックスの再作成**: データを大量に追加した場合、検索速度向上のためにインデックス（IVFFlat / HNSW）を再ビルドします。

## 2. 環境変数一覧

### 2.1 バックエンド (FastAPI)

| 変数名 | 必須 | 説明 | 設定例 |
|---|---|---|---|
| DATABASE_URL | 必須 | PostgreSQL 接続文字列 | `postgresql://app_user:pwd@/rag_chatbot?host=/cloudsql/project:region:instance` (本番) / `postgresql://postgres:pwd@db:5432/rag_chatbot` (ローカル) |
| GEMINI_API_KEY | 任意 | Google AI Studio 用の API キー (Vertex AI を使用しない場合のみ必須) | `AIzaSy...` |
| GEMINI_MODEL | 任意 | デフォルトで使用する LLM モデル | `gemini-2.5-flash` |
| GEMINI_FALLBACK_MODELS | 任意 | LLM 高負荷時のフォールバックモデル | `gemini-2.5-flash-lite` |
| USE_VERTEX_AI | 必須 | Vertex AI を優先して使用するかどうか | `true` (本番・検証) / `false` (ローカル開発用) |
| GCP_PROJECT_ID | 任意 | GCP プロジェクト ID (Vertex AI 使用時必須) | `rag-chatbot-prod` |
| GCP_LOCATION | 任意 | GCP リージョン (Vertex AI 使用時必須) | `asia-northeast1` |
| LLM_PROVIDER | 任意 | 使用する LLM プロバイダー | `gemini` |
| TOP_K_DEFAULT | 任意 | 検索時のデフォルト取得チャンク数 | `5` |
| CORS_ORIGINS | 任意 | CORS 許可オリジン | `https://frontend-url.run.app,http://localhost:3000` |

### 2.2 フロントエンド (Next.js)

| 変数名 | 必須 | 説明 | 例 |
|---|---|---|---|
| NEXT_PUBLIC_API_BASE_URL | 必須 | バックエンド API のエンドポイント URL | `https://rag-chatbot-backend-xxx-an.a.run.app` |

## 3. 監視・ログ・アラート

Google Cloud が提供する **Cloud Operations (旧 Stackdriver)** を利用します。

### 3.1 ログ確認 (Cloud Logging)

アプリケーションのログやエラー追跡は、Cloud Console の **Cloud Logging** で一元管理されます。

- **バックエンドログ**: Cloud Run の `rag-chatbot-backend` ログを確認
- **データベースログ**: Cloud SQL の `rag-db` ログを確認
- **クエリフィルタ例 (エラーログの抽出)**:
  ```
  resource.type="cloud_run_revision"
  severity>=ERROR
  ```

### 3.2 可用性監視 (Cloud Monitoring / Uptime Checks)

システムの稼働状況を監視するために、Cloud Monitoring で **Uptime Checks (稼働時間チェック)** を設定することを推奨します。

- **ターゲット**: バックエンドの `https://<backend-url>/health` エンドポイント
- **頻度**: 1 分〜5 分ごと
- **アラート**: 連続して 3 回以上チェックが失敗した場合、メールや Slack にアラート通知するポリシーを作成します。

## 4. トラブルシューティング

| 症状 | 考えられる原因 | 対策 |
|---|---|---|
| API `/health` で `database: error` | 1. Cloud SQL インスタンスが停止している<br>2. サービスアカウントの権限不足<br>3. 接続文字列の間違い | 1. GCP コンソールで Cloud SQL の稼働状況を確認<br>2. Cloud Run に設定されたサービスアカウントが `Cloud SQL クライアント` ロールを持っているか確認<br>3. `DATABASE_URL` 設定を確認 |
| API `/health` で `llm: error` | 1. `GCP_PROJECT_ID` が間違っている<br>2. Vertex AI API が有効化されていない<br>3. サービスアカウントに `Vertex AI ユーザー` ロールがない | 1. 環境変数の設定を確認<br>2. GCP プロジェクトで API を有効化<br>3. サービスアカウントの IAM ロール設定を確認 |
| 検索でエラーになる | インジェスト時のベクトル次元が合っていない | DB の `document_chunks.embedding` が `VECTOR(768)` になっているか確認し、合っていない場合はスキーマを再作成してデータを再インジェストする |
